**Date:** 2026-04-29
**Build:** Hangs-Local · Debug-Local · iPhone 17 Pro sim (918FD36A-8869-48F8-A1F8-3047CB122582)
**Tree:** main @ 6648ec1 + WIP in AnswerConfirmationView.swift, QuizViewModel.swift, UITestSupport.swift, NetworkService.swift, QuestionView.swift
**Driver:** Claude (XcodeBuildMCP UI automation + curl HTTP listener)

## VERDICT: FAIL — auto-confirm race; agent could not land the Edit-button tap before the 10 s timer fired

Three attempts. Each time the auto-confirm timer in `AnswerConfirmationView`
fired `confirmAnswer()` → `resubmitAnswer(_:suppressAudio:)` before the
agent's Edit-button coordinate tap landed on the sheet. Because `Edit` was
never entered, `beginEditingTranscript()` never ran, the auto-confirm timer
was never cancelled, and the scenario's central path (edit → type →
re-record) was never exercised.

This is the same harness limitation that RS-05 documented for cancel-from-
sheet flows. The agent's snapshot/tap round-trip exceeds the remaining
auto-confirm budget by the time the sheet has rendered, even when the
tap is sent without an intermediate snapshot.

The good news is bug-A from issue-19 is now fully fixed in the WIP tree
(see "Side observations" below) — RS-07 just hits a different blocker.

## Why the agent lost the race (3× same pattern)

Same root cause as RS-05's documented race, with a tightened understanding:

1. **The 10 s auto-confirm budget starts at `recording → processing`, not at curl.**
   `MockSTT` auto-emits a committed-transcript event near-instantly when the
   user taps mic (~5 ms in attempt 3, before any curl injection). So the
   sheet renders within milliseconds of mic-tap, and the curl is a no-op.
2. **Snapshot+tap round-trip is ~3–5 s.** With the sheet up and the agent's
   first action being a snapshot to find the Edit button (or to confirm the
   sheet is renderable for tapping), the snapshot returns ~5 s into the 10 s
   budget. By the time the tap is sent and lands, the budget is exhausted or
   the sheet's rendered position has drifted from the agent's cached coords.
3. **`label "Edit answer"` does not match in axe.** The snapshot exposes
   `AXLabel = "Edit answer"` on the pencil, but `tap({label: "Edit answer"})`
   returns "No accessibility element matched". Likely because the parent
   `confirmation.state.transcript` accessibilityIdentifier is shadowing
   children (RS-05 followup #2 — same shadowing bug). Coordinate fallback is
   the only option, and coordinates are vulnerable to detent animation timing.
4. **Coordinate (356, 456) was correct in the static snapshot but not always
   when tapped.** When the sheet animation is mid-flight at the moment of
   tap, the Edit button's hit region may be off-target. Without a way to
   confirm tap-landing without further latency, attempts compound.

## What would let RS-07 (and RS-06, RS-08) pass cleanly

These are the same followups RS-05 documented; RS-07 is fresh evidence the
problem is structural to the harness, not specific to the cancel scenario:

1. **Lengthen the auto-confirm timer in `--ui-test` mode** (e.g. 30 s) so
   the agent has time to react. Smallest behavior change. *(Recommended.)*
2. **Disable auto-confirm entirely under `--ui-test`** and let scenarios
   drive both confirm and re-record explicitly.
3. **Restore per-button accessibility identifiers** in the transcript
   branch (currently shadowed by `confirmation.state.transcript`). With
   `tap({id: "confirmation.edit"})` working, the snapshot step would
   become unnecessary and the round-trip would compress to a single tap.

Either (1) or (2) unblocks RS-06/07/08. (3) tightens all RS-NN runs and
should ride along.

## Side observations (independent of RS-07 verdict)

1. **bug-A from issue-19 is FIXED in the WIP tree.** All three attempts
   show `processing → showingResult` succeeding via auto-confirm with no
   `❌ REJECTED transition` line and no `❌ No evaluation in response`
   line. `UITestSupport.swift:43` wires
   `network.mockTextInputResponse = QuizResponse.previewAnswerCorrect`
   (issue-19 fix C1). The `processing → processing` rejection is also
   gone — either the redundant transition was guarded (issue-19 fix A1)
   or the transition table was extended. Worth verifying which by reading
   the WIP `QuizViewModel.swift`, but the user-facing symptom is healed.
2. **NEW BUG: questionId mismatch on every text-input answer.**
   `MISMATCH: evaluation.questionId=q_preview_1 != currentQuestion.id=q_preview_123`
   (and `…q_preview_456` for Q2). The `previewAnswerCorrect` fixture's
   `questionId` is hardcoded to `q_preview_1`, but the active question on
   each turn has its own id (`q_preview_123`, `q_preview_456`, etc).
   `handleQuizResponse` logs the mismatch as an error, then **applies the
   evaluation anyway**, so the user sees "NAILED IT" with the wrong
   question's evaluation grafted onto the current question. Not a crash,
   not user-visible in this fixture, but the log says we're shipping the
   wrong evaluation when the mock is wired this way. Two fix shapes:
   (a) make the mock build a synthetic `Evaluation` whose `questionId`
   matches the active question (issue-19 (C2) — query-aware mock); or
   (b) add a guard in `handleQuizResponse` that rejects mismatched
   evaluations rather than logging-and-applying. (a) is the right fix
   for the test harness; (b) is the right fix for production.
3. **App stayed alive throughout.** No `EXC_*`, no crash. State machine
   never entered `error` (because bug-A's mock-no-evaluation symptom is
   gone). Reset-to-home never happened.
4. **Auto-confirm tightened from 15 s (RS-05 era) to ~10 s.** RS-05's run
   reported a 10.6 s sheet-to-fire window; this run shows 10.4 s
   consistently across three attempts. If anything has changed, it's
   tightened the window further.

## Followups (separate from RS-07 verdict)

1. **Auto-confirm race blocks RS-06/07/08 (and RS-05).** This is now the
   single most blocking harness issue. Pick (1) or (2) from "What would
   let RS-07 pass cleanly" and ship it. Without it, every edit-flow
   scenario will FAIL on the same race.
2. **questionId mismatch in `previewAnswerCorrect`.** New issue surfaced
   here; deserves its own ticket (`docs/issues/issue-NN-…`). Not a
   blocker for RS-07, but worth fixing alongside the mock-evaluation
   work because the same fixture file is in scope.
3. **Re-record / Confirm / Edit buttons in transcript branch should
   expose their own accessibility identifiers** — same RS-05 followup
   #2; still open. Until fixed, all RS-NN scenarios that target buttons
   on the transcript branch must use coordinate fallback.
