**Driver:** Claude (XcodeBuildMCP UI automation + curl HTTP listener)

## VERDICT: FAIL — auto-confirm race; bug-A from RS-01 followups blocks this scenario

The auto-confirm timer in `AnswerConfirmationView` (transcript branch) fired
`resubmitAnswer(_:suppressAudio:)` ~10.6 s after the sheet appeared and beat
the agent's cancel/Re-record tap. The illegal `processing → processing`
transition was rejected, the mock backend response had no evaluation, and
the state machine routed to `error`, then `error → idle` (reset to home),
exactly as documented in the RS-01/RS-02/RS-04 followups.

The cancel path that RS-05 is supposed to exercise was therefore never
reached: the assertion point ("after dismissal, `question.state` ==
`askingQuestion`, no confirmation sheet visible") cannot be evaluated
honestly because the dismissal happened via the auto-confirm bug, not via
the cancel/Re-record button.

Per the run brief: "If the auto-confirm fires before your cancel tap,
report VERDICT: FAIL with reason 'auto-confirm race; bug-A from RS-01
followups blocks this scenario' and stop." — stopped here without retrying.

## Why the agent lost the race

Two ~5 s windows added up to a single ~10 s auto-confirm budget that the
agent could not realistically beat under the current snapshot/tap latency:

1. **Mock auto-emit + sheet render:** mock fires `committedTranscript` near-
   instantly when curl injects (or even on `startStreaming`), so the sheet
   appears within hundreds of ms of the mic tap. The auto-confirm timer
   starts at sheet-appear, with a 10 s budget per the "auto-confirming in N
   seconds" label.
2. **Snapshot → response → tap latency:** the agent must snapshot to find
   the cancel target, encode the snapshot result back through MCP (large
   JSON), decide which control to tap, then issue the tap call. In this
   run the snapshot returned with the button label already at "5 seconds"
   — i.e. half the budget had already burned before the agent saw the
   sheet at all. The remaining ~5 s was not enough for `tap` to land
   before 15:29:13.620.

Even with the (113, 784) coordinate prepared from the just-returned
snapshot, the round-trip time exceeded the remaining budget.

## What would let RS-05 pass cleanly

Pick one (the followups in RS-01/RS-02/RS-04 already point at the same
bug-A; this run is fresh evidence that it now blocks the cancel scenario
end-to-end):

1. **Fix bug-A (preferred).** Auto-confirm should not call `resubmitAnswer`
   when the user has not edited the answer. `processing → processing`
   should not be attempted; if it is, it should be a legal no-op. The
   mock-network-no-evaluation case should not produce a user-facing
   `error` in `--ui-test` mode. Any of these three fixes alone would
   unblock RS-05.
2. **Lengthen the auto-confirm timer in `--ui-test` mode** (e.g. 30 s) so
   the agent has time to react. This is a UI-test seam, not a product
   change.
3. **Disable auto-confirm entirely under `--ui-test`** and let scenarios
   drive both confirm and cancel explicitly.
4. **Add a dedicated `confirmation.cancel` accessibility identifier on
   the Re-record button in the transcript branch** so coordinate fallback
   isn't required. Independent of the race itself, but would tighten the
   harness.

## Followups (separate from RS-05 verdict)

1. **Bug-A (auto-confirm `processing → processing` → error).** Now
   blocks four of five scenarios from cleanly exercising their intended
   path (RS-01 followup, RS-02 followup, RS-04 followup, and now RS-05
   *fully* — RS-01/02/04 still PASSed because their assertion point was
   reached before bug-A fired; RS-05's assertion point is *after* the
   cancel/Re-record tap, which the auto-confirm overtakes). This is the
   clear next thing to fix in the regression-harness path.
2. **Re-record / Confirm buttons in transcript branch should expose their
   own accessibility identifiers** instead of being shadowed by the
   parent `confirmation.state.transcript`. Restoring per-button
   identifiers would let `tap(id: "confirmation.reRecord")` work without
   coordinate fallback.
3. **`confirmation.cancel` is on the processing branch only** — RS-05's
   spec wording ("Tap `confirmation.cancel` if present, otherwise
   dismiss with `confirmation.reRecord`") is consistent with that, but
   the runner brief should state explicitly that in the transcript
   branch the dismissal target is always Re-record.
4. **Sentry SentryCppHelper.o ld warning** (`found architecture 'arm64',
   required architecture 'x86_64'`) is unrelated but noted (same as
   RS-01/RS-02/RS-04).
