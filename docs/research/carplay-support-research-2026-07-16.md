# CarPlay Support Research — #97 — CarPlay support

**Date:** 2026-07-16 · **Synthesized from:** codebase recon + 3 adversarially-verified web-research lanes (policy, tech, prior art). Feeds `docs/issues/issue-97-carplay-support.md`.

## Executive summary — feasibility verdict

**Technically feasible; commercially gated on one discretionary Apple approval with HIGH rejection risk.**

- **Architecture fit is unusually good.** Apple's new **Voice-Based Conversational** CarPlay category (`com.apple.developer.carplay-voice-based-conversation`, iOS 26.4, Feb 2026) mandates a voice-primary, turn-based speaking/listening interaction with minimal templated UI — structurally almost identical to Trubbo's existing quiz loop (TTS question → voice answer → evaluate → next). ChatGPT shipped under it in March 2026, proving the category is live for third parties ([9to5Mac](https://9to5mac.com/2026/02/18/ios-26-4-adds-support-for-voice-based-ai-apps-to-carplay/), [MacRumors](https://www.macrumors.com/2026/03/31/openai-chatgpt-carplay/), [OpenAI Help](https://help.openai.com/en/articles/20001153-using-chatgpt-in-carplay)).
- **Policy fit is the problem.** CarPlay has a closed category list with **no games/trivia/entertainment category**, and games are excluded platform-wide for driver-distraction reasons ([Apple CarPlay categories](https://developer.apple.com/carplay/), [CarPlay Developer Guide](https://developer.apple.com/download/files/CarPlay-Developer-Guide.pdf), [carplayhacks](https://blog.carplayhacks.com/carplay-games/)). Adversarial verification **refuted** the hopeful claim that a trivia game "could plausibly qualify" under the voice-conversational category — treat approval-as-a-game as a non-foundation. The only credible pitch is a reframed "hands-free voice companion/Q&A" request, with rejection a live outcome to plan for.
- **The fallback is already shipped.** Trubbo's audio already routes to car speakers via standard `AVAudioSession` playback over Bluetooth/CarPlay audio with **zero entitlement** ([Apple audio session docs](https://developer.apple.com/documentation/avfaudio/handling-audio-interruptions)); a rejection costs the head-unit screen, not the in-car product.
- **One make-or-break technical unknown:** live STT quality/echo through the **car's mic** while CarPlay is connected is unverified by any first-hand report (verdict: *uncertain*); OpenAI's own ChatGPT CarPlay docs admit quality "depends on your car's microphones, speakers, and echo cancellation" ([OpenAI Help](https://help.openai.com/en/articles/20001153-using-chatgpt-in-carplay)). Requires physical head-unit validation; CarPlay simulators cannot test acoustics ([Apple: Using the CarPlay Simulator](https://developer.apple.com/documentation/carplay/using-the-carplay-simulator)).

**Recommendation:** pursue Voice-Based Conversational entitlement (file early, founder-gated, expect no SLA), build the CarPlay scene behind compile-flag + config isolation so mainline TestFlight is mechanically untouched, and sequence all agent work against the CarPlay simulator so the Apple wait never blocks progress.

## Entitlement / category analysis

**Category list (closed set, iOS 26.4 era):** Audio, Communication (messaging/VoIP), Navigation, EV Charging, Fueling, Parking, Public Safety, Quick Food Ordering, Driving Task, Voice-Based Conversational, plus Widgets/Live Activities and parked-only Video. One entitlement = one category = only that category's templates ([developer.apple.com/carplay](https://developer.apple.com/carplay/), [Requesting CarPlay Entitlements](https://developer.apple.com/documentation/carplay/requesting-carplay-entitlements), [CarPlay Developer Guide PDF](https://developer.apple.com/download/files/CarPlay-Developer-Guide.pdf)).

| Candidate | Fit | Notes |
|---|---|---|
| **Voice-Based Conversational** (`carplay-voice-based-conversation`, iOS 26.4+) | **Best — pursue** | Voice must be primary modality at launch and throughout; mandatory `CPVoiceControlTemplate` for listening/processing feedback; ≤3 template levels incl. root; no wake-word/steering-wheel invocation (manual launch only); audio session open only while voice features active; cannot control vehicle/phone ([MacRumors](https://www.macrumors.com/2026/02/18/ios-26-4-carplay-support/), [AppleInsider](https://appleinsider.com/articles/26/02/18/ai-agents-are-coming-to-carplay-but-theyre-not-getting-the-keys), [CPVoiceControlTemplate](https://developer.apple.com/documentation/carplay/cpvoicecontroltemplate)). Apple frames it as **task-driven services**, launch cohort is AI assistants (ChatGPT, Claude, Gemini, Google Meet) — a trivia game as-pitched is outside the stated scope. |
| Driving Task | Weak — rejected | "Tasks people need to do while driving" (car accessories, cab rides); 2 templates ≤iOS 26.3 / 3 on 26.4+; explicitly excludes non-driving tasks ([CarPlay Developer Guide](https://developer.apple.com/download/files/CarPlay-Developer-Guide.pdf), [developer.apple.com/carplay](https://developer.apple.com/carplay/)). |
| Audio | No — tertiary fallback only | Defined around continuous media playback (music/podcasts/radio) via now-playing/browse templates, not interactive Q&A with mic input ([carplay-audio entitlement](https://developer.apple.com/documentation/bundleresources/entitlements/com.apple.developer.carplay-audio), [newly.app guide](https://newly.app/how-to/carplay-entitlement)). Could theoretically host a passive Now-Playing quiz-progress display if voice-conversational is denied. |
| Games / Entertainment | **Does not exist** | No such category; games excluded from CarPlay entirely ([developer.apple.com/carplay](https://developer.apple.com/carplay/), [carplayhacks](https://blog.carplayhacks.com/carplay-games/)). |

**Request process & timeline:** filed at developer.apple.com/contact/carplay/ with bundle ID, single category, and a driving-safety/usefulness justification; individual case-by-case Apple review, **no published SLA**; documented turnarounds range from days to many months with prolonged silence common (one case "∞ days and counting"; a related DriverKit entitlement took 445 days); cited rejection reasons: doesn't clearly fit one category, unsafe for driving, app not substantively built ([newly.app](https://newly.app/how-to/carplay-entitlement), [mjtsai](https://mjtsai.com/blog/2022/09/08/apple-still-sitting-on-entitlement-requests/), [Apple forums](https://developer.apple.com/forums/thread/726791)). No 2026 evidence of improved turnaround. Trubbo is a real shipped TestFlight app, so "barely exists" shouldn't apply — but the "fit one category" and "game" objections are live.

## Technical architecture on CarPlay

**Scene model.** CarPlay UI is a second `UIScene` (`CPTemplateApplicationScene` + `CPTemplateApplicationSceneDelegate`) declared in Info.plist under `CPTemplateApplicationSceneSessionRoleApplication` — *not* the external-display role, a documented misconfiguration trap ([CPTemplateApplicationScene](https://developer.apple.com/documentation/carplay/cptemplateapplicationscene), [Apple forums](https://developer.apple.com/forums/thread/719752)). It runs in the **same process and binary** as the phone UI, sharing all services and state — isolation from MVP builds must therefore be build-time (config/flag/entitlement), never "it's a separate app". The CarPlay scene stays foreground while the phone is locked/backgrounded; conversely, connecting while already locked can leave the app effectively backgrounded until first unlock ([sentry-cocoa #8317](https://github.com/getsentry/sentry-cocoa/issues/8317), [Apple forums](https://developer.apple.com/forums/thread/132332)).

**Templates.** Voice-conversational apps get `CPVoiceControlTemplate` (listening/processing states, up to 4 action buttons) and a 3-level depth cap. List limits are **dynamic per head unit**: `CPListTemplate.maximumItemCount` observed from 12 to 500 — query at runtime, design for a single-digit worst case; `CPGridTemplate` effectively shows 8 items ([maximumItemCount](https://developer.apple.com/documentation/carplay/cplisttemplate/maximumitemcount), [Apple forums](https://developer.apple.com/forums/thread/667879)).

**Audio.** Car-speaker playback needs no entitlement — any foreground app's audio follows the system route (Bluetooth A2DP / CarPlay audio channel) ([Apple audio session guide](https://developer.apple.com/library/archive/documentation/Audio/Conceptual/AudioSessionProgrammingGuide/HandlingAudioInterruptions/HandlingAudioInterruptions.html)). This matches Trubbo's current in-car behavior. Voice-conversational apps configure `.playAndRecord` and must deactivate the session between voice turns ([WWDC26 "Rev up your CarPlay app"](https://developer.apple.com/videos/play/wwdc2026/212/)) — a lifecycle change vs. Trubbo's continuously-configured session in `apps/ios-app/Hangs/Hangs/Services/AudioService.swift`.

**Microphone / speech (the open risk).** No first-hand public report confirms SpeechAnalyzer/SFSpeechRecognizer accuracy or echo behavior through a CarPlay head unit's mic. Known physics: Bluetooth HFP is 8 kHz narrowband unless the head unit supports wideband, degrading recognition ([Apple QA1799](https://developer.apple.com/library/archive/qa/qa1799/_index.html)); iOS 26's `bluetoothHighQualityRecording` is AirPods-only and falls back to HFP for car mics ([WWDC25 session 251](https://developer.apple.com/videos/play/wwdc2025/251/), [API docs](https://developer.apple.com/documentation/avfaudio/avaudiosession/categoryoptions-swift.struct/bluetoothhighqualityrecording)); route descriptions expose whether the car does hardware echo cancellation so the app can avoid double-processing. ChatGPT's production CarPlay docs confirm quality varies by vehicle ([OpenAI Help](https://help.openai.com/en/articles/20001153-using-chatgpt-in-carplay)). **Fallback design if car-mic STT is unacceptable: keep the phone mic as input while CarPlay handles output/UI.**

**Testing.** Two entitlement-wait-independent rigs: Xcode's iOS Simulator CarPlay display, and the standalone CarPlay Simulator.app (Additional Tools) driving a real USB-connected iPhone ([Apple docs](https://developer.apple.com/documentation/carplay/using-the-carplay-simulator), [createwithswift](https://www.createwithswift.com/testing-apps-with-an-iphone-and-the-carplay-simulator/)). Apple explicitly notes vehicle acoustics can only be heard in a real car — mic/echo validation is physical-car-only.

**Trubbo-specific blockers (from recon, repo paths):** single `QuizViewModel` is owned by `ContentView` as `@StateObject` with only a `weak` back-ref in `AppState` (`apps/ios-app/Hangs/Hangs/ContentView.swift`, `Utilities/AppState.swift`) — a CarPlay scene today would spawn a second divergent quiz session; scene-phase handling tears down the mic on any `.background` transition (`QuizViewModel+ScenePhase.swift`) which would kill a CarPlay session on phone lock; voice-command screen taxonomy (`QuizViewModel+CommandListener.swift`) is phone-shaped; `fastlane/Gymfile` pins `Hangs-Prod`/`Release-Prod` and match runs `readonly: true`, so a missing entitlement in the match AppStore profile hard-fails rather than silently ships.

## Prior art

- **No native CarPlay trivia app exists as far as any source shows.** CarTrivia (closest competitor) is App Store category "Trivia", no CarPlay badge; its "works great with CarPlay" marketing describes Bluetooth audio passthrough + Siri launch ([cartrivia.app](https://cartrivia.app/), [App Store](https://apps.apple.com/us/app/id6757024865)). Drive.fm/Drivetime likewise phone-only audio (repo doc `docs/research/competitive-analysis-voice-driving-trivia-2026-06-27.md`; that doc's filled-circle "native CarPlay" mark for CarTrivia should be footnoted as unsupported). Pimsleur's "CarPlay experience" is consistent with passive Audio-category playback ([App Store](https://apps.apple.com/us/app/pimsleur-language-learning/id1405735469)).
- **Voice-conversational launch cohort:** ChatGPT (shipped Mar 2026), Claude/Gemini rolling out, Google Meet — all assistants, no games ([AppleInsider](https://appleinsider.com/articles/26/03/31/openais-chatgpt-now-available-hands-free-via-carplay-with-ios-264), [9to5Mac](https://9to5mac.com/2026/04/03/carplay-got-two-brand-new-apps-this-week-that-are-big-iphone-hits/)).
- **Cross-platform demand precedent (not Apple-approval precedent):** Android Auto once had official Assistant-powered voice trivia games (pre-2021) ([9to5Google](https://9to5google.com/2021/07/16/android-auto-gamesnacks-minigames/)); its parked-only GameSnacks layer was removed ~Oct 2025 ([9to5Google](https://9to5google.com/2025/10/06/android-auto-seems-to-be-removing-its-gamesnacks-collection/)). Amazon actively promotes trivia skills for Echo Auto ([Amazon dev blog](https://developer.amazon.com/en-US/blogs/alexa/alexa-auto/2019/11/automotive-skills-ways-alexa-can-make-your-drive-easier-more-fun-and-more-relaxing)). Voice trivia in cars is wanted and viable — Apple is the outlier gatekeeper.
- Per `docs/research/competitive-analysis-voice-driving-trivia-2026-06-27.md`, a native audio-first CarPlay presence is genuine competitive white space precisely *because* Apple blocks screen-based games.

## Risks

1. **Entitlement rejection or indefinite silence** — discretionary, no SLA, and a trivia game as-pitched is outside every category's stated scope. Mitigation: reframed pitch, file early, fallback = today's zero-entitlement audio passthrough.
2. **Car-mic STT quality/echo unverified** (T3) — could invalidate the native listening loop; fallback design keeps phone mic as input. Only testable in a physical car.
3. **Shared-binary leak risk** — CarPlay scene ships in the same target as MVP; an ungated Info.plist scene-manifest entry pointing at compiled-out code is a crash risk in Release-Prod. Must gate the manifest in lockstep with the Swift flag and verify by smoke build.
4. **Match profile regen touches the shared distribution cert** used by every mainline TestFlight build (per memory `project_ios_capability_profile_regen`) — founder-gated, must be done deliberately.
5. **Source fragility:** Apple's HIG page and the Developer Guide PDF resisted direct fetch; several requirement details rest on convergent secondary press (MacRumors/AppleInsider/9to5Mac) rather than verbatim Apple clause text. A manual read of the June 2026 guide PDF is advised before the entitlement request is worded.
6. **Category is 5 months old** — Apple could broaden or narrow eligibility before/while our request is in review.
