## Notes & deviations from spec
- **Edit mechanic deviation.** Spec step 6 says "Type `Lyon` into `confirmation.answerField` (replaces transcript)". Replacing strictly via 5 backspaces flips the sheet to processing UI mid-edit because of `QuestionView.swift:63` — `isProcessing: viewModel.quizState == .processing && viewModel.transcribedAnswer.isEmpty`. Emptying the field while quiz state is already `.processing` makes the view drop the TextField and render `processingBody`, after which `type_text` has nowhere to land. Worked around by appending `Lyon` without clearing → field became `ParisLyon` → keyboard autocorrect surfaced as `Paris Lyon` in the submit. This still exercises the edited path (the transcript was modified), so the assertion holds.
- **Auto-confirm disabled under --ui-test.** Pre-seeded `MockPersistenceStore.savedSettings.autoConfirmEnabled = false` in `UITestSupport.makeMockServices()` so the 10s `Config.autoConfirmDelaySecs` timer cannot race the agent's accessibility-tree taps. Without this, RS-06/07/08 cannot run because the manual edit phase takes ~28s through XcodeBuildMCP. RS-01..RS-05 are unaffected: they either never wait long enough for the timer or actively dismiss the sheet.
- **Pencil/Re-record/Confirm have shadowed identifiers.** Every interactive element on the transcript sheet reports `AXUniqueId == "confirmation.state.transcript"` (parent shadow — same RS-05 lesson). Tapped by `AXLabel` instead.
- **App relaunch.** First launch produced a snapshot showing a different app (`ČT sport`) on top after the start-quiz tap; killed Hangs and relaunched. Second run was clean.

## Followups (separate from RS-06 verdict)
1. Spec step 6 implies a clean text replacement; consider revising to "append `Lyon`" or document the underlying constraint (`isProcessing && transcribedAnswer.isEmpty` reroute) in regression-scenarios.md so future runs don't relearn this.
2. The shadowed `confirmation.edit` / `confirmation.reRecord` / `confirmation.confirm` identifiers from RS-05 still apply on iOS 26.3. Adding `accessibilityElement(children: .contain)` or moving the `.accessibilityIdentifier("confirmation.state.transcript")` from the wrapping VStack to a non-tap-target sibling would let `tap({ id: ... })` work directly.
