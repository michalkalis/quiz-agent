## Decisions

- **`QuestionRow` not `Question` for the ORM class** ‚Äî Pydantic's class is `Question` everywhere in the shared package; using the same name for the ORM class in the same module causes confusion. `QuestionRow` is explicit at the seam.
- **`append_step` uses a single `UPDATE ‚Ä¶ RETURNING`** ‚Äî the event_id is `jsonb_array_length(step_log)` evaluated in the SET RHS (reads OLD row) and equivalent to `RETURNING jsonb_array_length(step_log) - 1` (reads NEW row). Concurrent writers serialize on the row lock ‚Äî no Python-level read-modify-write. Matches R9 requirement verbatim.
- **Circular FK handled with `use_alter=True`** ‚Äî `generation_orders.job_id` / `pack_id` FK back to tables that themselves FK to orders. Migration creates orders first without those FKs, then adds them via `op.create_foreign_key` after the target tables exist. SQLAlchemy `use_alter=True` on the mapped columns mirrors this in ORM metadata.
- **Status = VARCHAR + CHECK (not native enum)** ‚Äî adding a status value in Phase 2/3 via `ALTER TYPE ‚Ä¶ ADD VALUE` is forward-only and can't run in a transaction pre-PG12. CHECK is a simple Alembic `ALTER TABLE ‚Ä¶ DROP CONSTRAINT ‚Ä¶ ADD CONSTRAINT`. Aligns with plan.
- **Extra Pydantic `Question` fields added as ORM columns** ‚Äî the plan sketch isn't exhaustive (omits `created_by`, `reviewed_by`, `quality_ratings`, `media_url`, etc.). Storing them as columns rather than packing into `provenance` JSONB keeps the round-trip test value-equal without treating provenance as a junk drawer.
- **Float32 embedding caveat** ‚Äî pgvector stores `Vector(n)` as float32; round-trip test compares embedding separately via `pytest.approx(abs=1e-6)` and clears it before model-level `==` check.
