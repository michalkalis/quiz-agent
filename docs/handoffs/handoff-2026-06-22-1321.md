## Decisions

- **Reuse the `LEGACY_USER_ID_GRACE` window for the quiz-agent bearer gate — do NOT hard-401 on no-bearer.** This is the load-bearing call. #60 already uses this flag (default `"on"`) so the freemium fix on `POST /sessions` didn't break the currently-deployed pre-auth iOS build. The #65 high-cost endpoints have the *same* coupling, so `require_authenticated_subject` mirrors `resolve_session_subject` exactly: a *present* bearer must verify (else 401), a *missing* bearer passes only during grace. Deploying quiz-agent is therefore **non-breaking** even before iOS auth ships; the hard gate activates when grace is flipped off (post-migration). A naive "no bearer → 401" would break the live app on deploy — rejected.
- **Gate the *whole* quiz-pack-api `/web` and `/api/v1` questions routers at router level** (`dependencies=[Depends(require_admin)]`). Verified safe: **no test and no script hits those HTTP paths** — the only `include_router`/HTTP tests target the **orders** router (`/v1/orders`, keeps its own `X-StoreKit-JWS` auth — **do NOT gate it**), and the generation/verify "http-mock" tests drive `AdvancedQuestionGenerator`/`FactVerifier` **services directly**, never the routes. Import scripts use storage directly, not HTTP.
- **Admin auth = `X-Admin-Key` header OR HTTP Basic** (one `require_admin` dep). Header matches quiz-agent (`misc.py:67`) for scripts/agents; Basic (any username, password = `ADMIN_API_KEY`, signalled by `WWW-Authenticate: Basic`) keeps the `/web` admin UI openable in a browser — a header-only guard would lock the founder out of their own UI. **Flag for founder confirmation.**
- **Fail closed when `ADMIN_API_KEY` is unset → 503** (matches #60's "Auth unavailable" pattern). Means local dev must set a dev key to use the admin UI; prod must set the Fly secret or everything 503s. Safe-by-default; avoids the exact ship-it-open footgun #65 is about.
- **Defer adding `slowapi` to quiz-pack-api** (the generation rate-limit in the #65 *recommendation*, NOT in `## Acceptance`). Once the route requires an admin key, anonymous flooding is impossible; pulling a whole rate-limit framework into a second app for one decorator is over-footprint (Rule #1). Residual risk (a *leaked* admin key draining gpt-4o) is real but secondary — **flag for founder** as defense-in-depth they can opt into.
- **App Attest guard logs loud, doesn't refuse** — a hard boot-fail on a misconfig could take down prod; a loud `logger.error` in the deploy logs satisfies acceptance criterion 5 without that risk.
