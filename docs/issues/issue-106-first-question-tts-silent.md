## Investigation ‚Äî 2026-07-17

> *This was generated by AI during triage.*

**Ruled out: the known 401 cause.** The 2026-07-11 fix "question TTS silent ‚Äî attach bearer to audio downloads" (`033e6e2`) is an ancestor of `248d63d` ‚Äî it **is** in build 22. That failure mode (hard 401, logged + `Self.reportAudioFailure`) is not the culprit here; also the founder reports no error banner, just silence.

**Leading hypothesis (PROBABLE): first-play races the *cold* Bluetooth HFP/A2DP negotiation, and the app double-switches the audio-session category on top of it.**

Trace of `startNewQuiz` ‚Üí first TTS (`apps/ios-app/Hangs/Hangs/ViewModels/QuizViewModel.swift:588-639`):
1. `audioService.setupAudioSession(mode:)` ‚Äî `AudioService.swift:212-218` ‚Äî `setCategory(.playAndRecord, options: [...allowBluetoothHFP...])` then `setActive(true)`. This is the session's **first-ever activation**, which is what actually kicks off the car's Bluetooth profile handshake (HFP/A2DP). On a cold link this negotiation is known to take on the order of seconds on some head units.
2. Two awaited network round-trips follow (`createSession`, `startQuiz`, `QuizViewModel.swift:598-615`) ‚Äî some but not guaranteed-sufficient settle time.
3. First `playQuestionAudio` ‚Üí `playOpusAudio` ‚Üí `withPlaybackCategory` (`AudioService.swift:272-292`) does a **second** `setCategory(.playback, options:[.duckOthers,.interruptSpokenAudioAndMixWithOthers])` + `setActive(true)`, moments after step 1, potentially while the car's first handshake from step 1 is still settling.

Nothing in the code waits for the route to actually include the Bluetooth port before playing (no `routeChangeNotification` await, no delay) ‚Äî confirmed absent via grep across `AudioService.swift`. If CoreAudio treats the second category+activate as interrupting an in-flight route negotiation, the resulting AVPlayer can report `.playing` (KVO, `AudioService.swift:876-887`) with no exception thrown, while no audio actually reaches the car ‚Äî this matches the founder's report exactly: no error banner, "just" silence. For question 2+, the same `.playAndRecord ‚Üî .playback` toggle (`AudioService.swift:287,292,306,309`) is flipping between two *already-negotiated* profiles on a stable link, so it's fast and reliable ‚Äî consistent with "sometimes / first is worst" rather than "always."

**Same underlying mechanism as issue #104, not a duplicate.** #104 (CONFIRMED) already documents that `withPlaybackCategory` renegotiates the BT route on *every* utterance (`AudioService.swift:287`, commit `331c47c`, present in build 22) and that this flapping is real and reproducible for the mic-capture/call-mode symptoms. This issue is the TTS-output-side manifestation of the exact same flapping, with the extra first-play-specific wrinkle: question 1's category switch lands on top of a **cold** BT negotiation instead of a warm one, making silent failure most likely there. Fixing #104's flapping (hold one category for the whole quiz, or avoid the category swap entirely via `overrideOutputAudioPort`) would very likely also fix or substantially reduce this ‚Äî do not build a second, separate flapping fix here.

**Not fixed on current main (unshipped).** `withPlaybackCategory` and `setupAudioSession` are unchanged between `248d63d` and `main`; #100's changes (`a669e39`, interruption `.ended` reactivation) touch a different code path (interruption recovery, not the setup‚Üífirst-play window). This symptom is present in build 22 and still present on unshipped main.

**Ruled out:** `isMuted` default is `false` (`QuizSettings.swift:91,132`) with no odd decode path (`:170`) ‚Äî not a mute-state bug.
