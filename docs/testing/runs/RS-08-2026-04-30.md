## Edit-branch snapshot (proves the gap)
At step 6 the sheet contained only:
- `YOU SAID` (StaticText, `confirmation.state.transcript`)
- `confirmation.answerField` (TextField, `ParisBerlin`)
- `Re-record` button (`confirmation.state.transcript`)
- `Confirm answer` button (`confirmation.state.transcript`)
- `Toolbar` group (keyboard accessory: a single `Done` button that just unfocuses — does not undo the edit)

No `confirmation.cancel` accessibility id, no `Cancel` AXLabel, no SF symbol
`xmark` button — the affordance is genuinely missing in the edit branch.

In `Views/AnswerConfirmationView.swift`, `confirmation.cancel` exists only on
the **processing** branch (`processingBody`, line 215), wired to
`onCancel: { viewModel.cancelProcessing() }`. The transcript branch's edit
mode (`editableTranscript`, line 123) exposes only the keyboard `Done`
accessory, which dismisses focus but does not restore the pre-edit
transcript or revert state.

## Path to PASS

When this scenario is unblocked, the implementation needs:

1. **View:** add a Cancel control to `editableTranscript` (e.g., a leading
   `xmark.circle` Button, or a second toolbar item beside `Done`) with
   `accessibilityIdentifier("confirmation.cancel")` (or pick a distinct id
   like `confirmation.editCancel` to avoid collision with the
   processing-branch button).
2. **View-model:** new `cancelEditingTranscript()` that snapshots
   `transcribedAnswer` at the start of `beginEditingTranscript()` and
   restores it on cancel; sets `transcriptWasEdited = false`; **does not**
   transition state (sheet stays up).
3. **Wiring:** `QuestionView.swift` passes a new `onCancelEditing` closure
   into `AnswerConfirmationView`; the view's edit-branch button calls it.

After (1)+(2)+(3), re-run RS-08 and capture a new run report dated the
implementation day.

## Followups (separate from RS-08 verdict)
1. Decide whether the affordance lives in the keyboard toolbar (alongside
   `Done`) or as an inline button on the sheet itself. The latter is more
   discoverable for hands-free usage; the former conserves layout for the
   transcript display.
2. Avoid the `confirmation.cancel` id collision: today the same id labels
   the processing-branch "Cancel processing" button. If the edit-branch
   cancel reuses that id, RS-05 / future scenarios that disambiguate by id
   will pick the wrong target depending on which branch is rendered. A
   distinct id (`confirmation.editCancel`) is safer and matches the model
   intent ("cancel this edit", not "cancel processing").

VERDICT: FAIL — `confirmation.cancel` (or equivalent) is missing from the
edit branch; this scenario remains forward-looking until that affordance
ships.
