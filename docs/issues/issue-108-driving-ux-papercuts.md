## Symptom (founder car test, 2026-07-16)

Three items from a real in-car test (build 22, 2026-07-14, commit `248d63d`):

**A. Re-record doesn't auto-start.** On the answer-confirmation sheet, tapping "Re-record" closes the modal but recording does not begin — founder has to press the mic/record button again manually. Expected: tapping re-record should start a new recording immediately.

**B. Timer-in-button (nice-to-have, founder asked to record it anyway).** When a countdown is on screen (time-to-answer, or auto-advance-to-next-question), show it directly inside the primary CTA button — a progress bar draining right-to-left within the button — instead of (or alongside) a separate bar.

**C. Screen dims/locks during quiz.** On both the quiz screen and the result screen, the iPhone display kept auto-dimming during play. For a hands-free driving app the screen must stay awake for the whole active session.

## Investigation — 2026-07-17

> *This was generated by AI during triage.*

### A. Re-record — CONFIRMED, by-design behavior that contradicts the ask

`rerecordAnswer()` (`apps/ios-app/Hangs/Hangs/ViewModels/QuizViewModel+Recording.swift:569-579`) does **not** call `startRecording()`. It transitions back to `.askingQuestion` and calls `startAnswerTimer(extraSeconds: 10, bypassRerecord: true)` — a fresh full countdown (`answerTimeLimit + 10s`) that only auto-starts recording once it **fully expires** (`QuizViewModel+Timers.swift:79-108`, the `await self.startRecording()` at line 105). Separately, `isRerecording = true` explicitly suppresses the normal auto-record/thinking-time path (`QuizViewModel+Audio.swift:98,128`, `QuizViewModel.swift:1206`, all gated `&& !isRerecording`), so the fast auto-record route is deliberately skipped for re-records too.
Net effect: after tapping "Re-record" the founder lands on a screen with a manual mic button (`QuestionView.swift:478-500`, `question.record`) and a slowly-counting timer — not an active recording — matching his report exactly. This is intentional-looking code (comment: "Return to ready state, not recording") but conflicts with the founder's expectation; not covered by #100 (that issue doesn't touch re-record) and unrelated to unshipped-vs-main — this behavior is unchanged on current main.

### B. Timer-in-button — HYPOTHESIS / design gap, not started

No in-button progress bar exists anywhere. Current pattern is a **separate** bar below the CTA row: `AnswerConfirmationView.swift:109-112` (`countdownBar`, defined `:189-`) and the analogous `ResultView.swift:51` / `:184-` auto-advance bar, both `GeometryReader`-based. `HangsPrimaryButton` (`Views/Components/Hangs/HangsButton.swift:13-50`) is a plain pill (label + optional icon, no progress layer) — moving the countdown inside it is new component work, not a fix to existing broken code. Two on-screen countdowns map to this: the confirm-sheet auto-confirm bar and the result-screen auto-advance bar; `QuestionView.swift` also drives `thinkingTimeCountdown`/`answerTimerCountdown` display (`:196-198`) but currently as a separate strip, not button-hosted.

### C. Idle timer — CONFIRMED absent

`grep -ri "idleTimer"` across `apps/ios-app` returns **zero matches** — `UIApplication.shared.isIdleTimerDisabled` is never set anywhere in the app. Nothing in #100 (iOS driving-loop robustness) touches this either (its content is audio-interruption / concurrency fixes, not screen-wake). So this is a straight gap, present on both build 22 and current main — not something #100 already fixes.
`ContentView.swift:76-100` is the natural scope boundary: it already switches on `viewModel.quizState` (`.idle` / `.startingQuiz,.askingQuestion,.recording,.processing,.skipping` / `.showingResult` / `.finished`) and knows `viewModel.isMinimized`. An active session = any state other than `.idle`/`.finished` while not minimized.

## Proposed fix direction

**A.** In `rerecordAnswer()`, replace the countdown-then-record path with the same immediate-start used by the manual button (`cancelAnswerTimer()` + `await startRecording()`, mirroring `toggleRecording()`'s `.askingQuestion` branch, `QuizViewModel+Recording.swift:16-21`). Keep (or drop, founder call if it comes up) the "+10s bonus" framing — it becomes moot once recording starts immediately instead of after a countdown.

**B.** New component: a variant of `HangsPrimaryButton` (or a wrapper) that layers a right-to-left-draining fill/progress behind the label, driven by the same countdown values already computed (`autoConfirmCountdown`/`autoConfirmTotal` in the confirmation sheet, the result-screen equivalent). Replaces the separate `countdownBar` usages. **Sequence locked by founder decision above:** Pencil frame draft → founder sign-off → implementation. Keep the visual minimal (Waze-like fill inside the current pill shape).

**C.** Add a small idle-timer controller (e.g. a `.onChange(of: viewModel.quizState)` / computed-state modifier in `ContentView.swift`) that sets `UIApplication.shared.isIdleTimerDisabled = true` while state ∉ {`.idle`, `.finished`} and not minimized, and `false` otherwise — including on `.onDisappear` of `ContentView` (app teardown safety net) so the flag never leaks true past an active session.
