## Decisions

- **Binding flow (the issue's step-2/step-3 text was self-contradictory on *when* the key binds to an identity).** Resolved to **one endpoint, two credential modes** on `anon-bootstrap`:
  - First launch → request carries `attestation`: verify → mint identity → bind key 1:1 → issue first refresh, **all in one DB transaction**. The attestation itself is the liveness proof (`counter==0`); no separate assertion needed first.
  - Re-bootstrap → request carries `assertion`: verify against the stored key (advances counter transactionally), re-issue tokens for the **already-bound** identity. An assertion can **never** mint a new identity.
  - Flag off / empty body → mints plainly (Part A client + dev/sim untouched). Flag on + no usable credential → 401. Flag on + verifier unconfigured (no `APP_ATTEST_APP_ID`) → 503 (fail-safe).
  - **Net invariant: one Secure-Enclave key = exactly one anonymous identity, for its lifetime.** This is the iOS contract 60.13 must implement.
- **Did not blind-code iOS 60.13.** App Attest can't run on the simulator, so the code is writable but not E2E-verifiable without a device; chose to freeze the contract + hand off rather than ship unverifiable Swift. (User confirmed they can test on device when needed — so writing + `mba` compile-check is a reasonable next move if desired.)
- **Two review-focus flags** left for the security reviewer (in the 60.12 note): (a) challenge-consume is a *separate* txn from the bootstrap txn — a consumed-then-rolled-back challenge is merely wasted, not a hole; (b) re-attesting an already-stored `key_id` → DB integrity error → 500 (anomalous client only, not special-cased).
