## Decisions

- **42.9 split into 42.9a + 42.9b**: the original single task assumed you could specify per-question type *before* generation. In reality the LLM picks the pattern during generation — type must be assigned *after* by inspecting `q.pattern`. 42.9a does post-gen tagging + fail-loud drop (no generator changes); 42.9b adds the prompt-side `possible_answers` emission instruction. Keeping them separate means 42.9a can land and be tested before touching the generator.
- **Pydantic Literal/validator added to 42.8**: `Question.type` is a free `str` with no enforcement. Adding a validator to 42.8 (pattern routing) rather than a separate task keeps it in the same atomic commit as the routing map it supports.
- **Evaluator routing is on `possible_answers`, not `type`**: the fast-path fires when `question.possible_answers` is truthy (`evaluator.py:77`). `type=text_multichoice` is the iOS signal to show `MCQOptionPicker`. These are separate concerns — plan now documents both clearly.
