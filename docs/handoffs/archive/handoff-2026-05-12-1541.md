## Decisions

- **ON CONFLICT DO UPDATE (not DO NOTHING)** â€” 1.6 used DO NOTHING because pending rows are pre-approval. 1.7 must overwrite `review_status='pending_review'` rows that 1.6 may have inserted for the same legacy id, otherwise the acceptance count would be wrong. Upsert targets only the 7 columns ChromaDB is authoritative for (`review_status, pack_id, question, embedding, embedding_model, embedding_dim, provenance`).
- **Import `_legacy_to_uuid` from 1.6 script** rather than redefining â€” guarantees identical UUID derivation for the same legacy id, preventing duplicate Postgres rows when a question existed in both pending.db and ChromaDB.
- **No re-embedding** â€” ChromaDB already caches `text-embedding-3-small` 1536-d vectors for all approved questions; copying them is free and exact.
- **Runbook (a) chosen over (b)** â€” `fly ssh sftp get` to local + run against `$PROD_DATABASE_URL`, same as 1.6. Approach (b) (`fly ssh console` inside quiz-agent with cross-app DATABASE_URL) rejected for infra friction on a one-shot migration. Documented as alternative in script docstring.
- **chromadb excluded from Dockerfile** â€” intentionally, same as the comment already in the Dockerfile. The script's `import chromadb` is guarded with a clear error if missing; it runs only locally or inside quiz-agent, never inside quiz-pack-api prod image.
