# Writing Agent Briefs

An agent brief is a structured section appended to a `docs/issues/issue-NN-*.md` file when it moves to `ready-for-agent`. It is the authoritative spec an autonomous agent will work from. The original issue body and discussion are context — the agent brief is the contract.

## Principles

### Durability over precision

The issue may sit in `ready-for-agent` for days or weeks. The codebase will change in the meantime. Write the brief so it stays useful even as files are renamed, moved, or refactored.

- **Do** describe interfaces, types, behavioral contracts (use CONTEXT.md vocabulary)
- **Do** name specific types, function signatures, or config shapes that the agent should look for or modify
- **Don't** reference exact file paths if avoidable — they go stale
- **Don't** reference line numbers
- **Don't** assume the current implementation structure will remain the same

(Some file paths are stable enough to reference — `apps/quiz-agent/`, `apps/ios-app/Hangs/Hangs/`, `docs/testing/regression-scenarios.md`. Use judgment.)

### Behavioral, not procedural

Describe **what** the system should do, not **how** to implement it. The agent will explore the codebase fresh and make its own implementation decisions.

- **Good:** "When the auto-confirm timer fires on an unedited transcript, it must not call `resubmitAnswer` — instead it should route through the same path as a user-initiated confirmation."
- **Bad:** "Open `QuizViewModel.swift` line 142 and change the function call."
- **Good:** "Streaming-STT confirmations should reach the same `confirmAnswer` path regardless of whether the user edited the transcript."
- **Bad:** "Add an if statement before the resubmit call."

### Complete acceptance criteria

The agent needs to know when it's done. Every brief must have concrete, testable acceptance criteria, each independently verifiable.

- **Good:** "RS-05 reaches its terminal assertion (state `evaluated`, evaluation surface populated) without going through a REJECTED `processing → processing` transition."
- **Bad:** "Auto-confirm should work correctly."

### Explicit scope boundaries

State what is out of scope. Prevents gold-plating or assumptions about adjacent features.

## Template

```markdown
## Agent Brief — <YYYY-MM-DD>

> *This was generated by AI during triage.*

**Category:** bug / enhancement
**Summary:** one-line description of what needs to happen

**Current behavior:**
What happens now. For bugs, this is the broken behavior.
For enhancements, this is the status quo the feature builds on.

**Desired behavior:**
What should happen after the agent's work is complete. Be specific
about edge cases and error conditions.

**Key interfaces / call paths:**
- `TypeName` — what needs to change and why
- `functionName()` return type — what it currently returns vs what it should return
- Specific call paths or transitions, named in CONTEXT.md vocabulary

**Acceptance criteria:**
- [ ] Specific, testable criterion 1 (ideally an existing or new RS-NN)
- [ ] Specific, testable criterion 2
- [ ] Specific, testable criterion 3

**Out of scope:**
- Thing that should NOT be changed or addressed
- Adjacent feature that might seem related but is separate

**Suggested feedback loop:**
What `/diagnose` Phase 1 loop the agent should reach for first
(e.g. "RS-05 via `/regression`", "pytest tests/<dir>/test_<x>.py", "curl
against `localhost:8002/...`").
```

## Examples

### Good agent brief (bug)

```markdown
## Agent Brief — 2026-04-29

> *This was generated by AI during triage.*

**Category:** bug
**Summary:** Auto-confirm of unedited transcript routes through `resubmitAnswer` and errors out

**Current behavior:**
~10s after the confirmation sheet appears with an unedited streaming-STT
transcript, the auto-confirm timer fires `confirmAnswer()`, which falls
into `resubmitAnswer(_:suppressAudio:)` because `pendingResponse == nil`
on that path. `resubmitAnswer` calls `transition(to: .processing)` while
state is already `.processing` — the transition is REJECTED. The mock
then surfaces "Could not evaluate your answer."

**Desired behavior:**
Streaming-STT auto-confirms (and user-initiated confirms of unedited
transcripts) reach the regular submit path, not the resubmit path. The
state machine never re-enters a state it's already in.

**Key interfaces / call paths:**
- `QuizViewModel.confirmAnswer()` — branching on edited vs unedited
- `QuizViewModel.resubmitAnswer(_:suppressAudio:)` — should be called
  only when the user actually edited the transcript
- `QuizState.validTransitions[.processing]` — does not include `.processing`,
  this invariant must remain
- `AnswerConfirmationView` auto-confirm timer — caller of `confirmAnswer()`

**Acceptance criteria:**
- [ ] RS-05 reaches its terminal assertion without REJECTED transitions in the log
- [ ] RS-05 evaluation surface is populated correctly
- [ ] A new unit test in `QuizViewModelTests` exercises the unedited
      auto-confirm path and asserts no REJECTED transitions
- [ ] User-edited transcript path still works (RS-edit-resubmit, write if missing)

**Out of scope:**
- Renaming `resubmitAnswer` (orthogonal cleanup, file separately)
- Mock evaluation behavior in `QuizResponse.previewStartQuiz` — keep as-is

**Suggested feedback loop:**
RS-05 via `/regression RS-05` against `Hangs-Local`. Augment with a unit
test once the fix is in.
```

### Good agent brief (enhancement)

```markdown
## Agent Brief — 2026-04-15

> *This was generated by AI during triage.*

**Category:** enhancement
**Summary:** Surface in-app a "Report wrong answer" widget that posts to the backend

**Current behavior:**
Users have no in-app channel for reporting question quality issues.
Wrong-answer reports must currently come via TestFlight feedback or
direct messages.

**Desired behavior:**
After each evaluated question, the user can flag the question as wrong /
unclear via a button on the result surface. The flag posts to the
backend with question ID, user-supplied reason (optional), and current
quiz session id.

**Key interfaces / call paths:**
- New backend endpoint accepting `{question_id, reason?, session_id}`
- iOS `NetworkService` gains a corresponding method
- Result-screen view appends a small "Report" affordance
- Persistence: SQLite ratings store already used for question scores —
  same mechanism extends naturally

**Acceptance criteria:**
- [ ] Endpoint round-trips a flag and persists to the ratings store
- [ ] iOS posts the flag and shows confirmation
- [ ] /verify-api passes (Codable struct sync)
- [ ] Backend pytest covers the new endpoint
- [ ] Feature flag-gated behind a `flags.enableReporting` so we can turn
      it off if abuse appears

**Out of scope:**
- Reviewer-side surface for triaging reported questions
- Bulk export of reports

**Suggested feedback loop:**
Backend: pytest. iOS: build + manual on simulator (no RS scenario yet).
```

### Bad agent brief

```markdown
## Agent Brief

**Summary:** Fix the resubmit bug

**What to do:**
The resubmit thing is broken. Look at QuizViewModel and fix it.
Around line 142.

**Files to change:**
- apps/ios-app/.../QuizViewModel.swift (line 142)
```

This is bad because:
- No category, no Triage line update
- Vague description
- References file paths and line numbers that go stale
- No acceptance criteria
- No scope boundaries
- No feedback loop hint
- No description of current vs desired behavior
