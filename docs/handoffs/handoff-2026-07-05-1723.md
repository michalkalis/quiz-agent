# Handoff 2026-07-05 — Review promptov: projekt + UI/UX (3 sessions)

**Kontext:** Zakladateľ stratil prehľad o projekte — veľa implementácií a rozhodnutí prebehlo autonómne (Opus) a chce zrušiť known aj unknown unknowns. Zároveň chce, aby sa UI/UX pre nové features schvaľovalo PRED implementáciou, nie po nej. Výstupom tejto prípravy sú 3 session prompty nižšie. Spúšťať v poradí 1 → 2 → 3, každý v čerstvej session. Žiadna session nič neimplementuje — všetko sú review/explainer/návrhy na rozhodnutie.

---

## Session 1 — Projektové review a explainer

```
Sprav mi kompletné review projektu ako explainer pre zakladateľa. Som product owner, veľa implementácií a rozhodnutí prebehlo autonómne bez môjho vedomého schválenia a chcem obnoviť prehľad. Nič neimplementuj — výstup je čisto informačný.

Pokry štyri oblasti:

1. **Explainer celého systému** — ako projekt dnes reálne funguje: backend (quiz-agent), quiz-pack-api (generovanie otázok), iOS appka, web-ui, zdieľané modely, infra (Fly.io, Sentry, CI, agent Mac "mba" + Ralph loop). Čo je nasadené v produkcii vs. len lokálne, čo je hotové vs. rozpracované. Vysvetľuj koncepčne, v jednoduchom jazyku, bez kódu.

2. **Kľúčové rozhodnutia** — prejdi git históriu, docs/issues/, docs/handoffs/ a docs/research/ a vytiahni VŠETKY podstatné architektonické, produktové a infra rozhodnutia — tie, ktoré spravil agent autonómne, tie, ktoré som spravil ja, aj spoločné. Pri každom: čo sa rozhodlo, kto rozhodol (agent / ja / spoločne), prečo, aké boli alternatívy, a či sa to dá ešte zvrátiť (a za akú cenu). Osobitne zvýrazni tie, ktoré neprešli mojím explicitným schválením a mal by som ich spätne potvrdiť alebo prehodnotiť.

3. **Review plánu a backlogu** — docs/todo/TODO.md, docs/issues/INDEX.md, docs/product/INDEX.md. Čo je naplánované, v akom stave, čo je sporné/riskantné, čo by som mal vidieť skôr, než sa to spustí (hlavne cez Ralph autonomný loop). Explicitne označ issues, ktoré majú UI/UX povrch — tie pôjdu do samostatného UI review v ďalšej session.

4. **Riziká a technický dlh** — známe slabiny, nedotiahnuté veci, veci označené ako hotové ale neoverené. Tvrdenia typu "hotové/otestované" z docs ber ako hypotézy — kľúčové si over prvoruko (spusti testy, over deploy stav), neopieraj sa len o to, čo tvrdia dokumenty.

Postup: použi workflow na orchestráciu — paralelní čítači po oblastiach (backend, iOS, pipeline, infra, docs/git história), potom syntéza; orchestráciu si navrhni sám podľa úlohy. Bulk čítanie nech beží na lacnejšom modeli (sonnet), syntéza a overovanie na hlavnom; load-bearing tvrdenia subagentov si over prvoruko.

Výstup: jeden HTML artefakt docs/artifacts/project-review-2026-07.html — vizuálny, dark theme, scannable (status boardy, badges, count dashboardy, collapsible detaily), podľa vzoru docs/artifacts/issues-visual-recap-2026-06-18.html. Na záver artefaktu sekcia "Rozhodnutia pre zakladateľa" — očíslovaný zoznam vecí, ku ktorým odo mňa potrebuješ stanovisko. V odpovedi mi daj `open <path>` a krátke zhrnutie top zistení v slovenčine.
```

## Session 2 — Review existujúceho UI/UX so zdôvodnením

```
Sprav review existujúceho UI/UX iOS appky (apps/ios-app, SwiftUI). Nepotrebujem screenshoty — appku si viem pozrieť v simulátore sám. Potrebujem ZDÔVODNENIE: prečo je každý prvok tam, kde je, a vyzerá tak, ako vyzerá. Nič neimplementuj.

Prejdi všetky obrazovky (home/settings, quiz flow, "Type your answer", result obrazovka, auth, ďalšie čo nájdeš) čítaním SwiftUI kódu. Použi workflow: paralelní čítači po obrazovkách (bulk čítanie na sonnete), potom hodnotenie a syntéza; orchestráciu si navrhni sám. Pre každú obrazovku:

- zoznam prvkov a ich rozloženie (textový popis / jednoduchá schéma, nie kód),
- pri každom významnom prvku zdôvodnenie: prečo táto pozícia, veľkosť, štýl — opri sa o iOS HIG, o náš primárny use-case (hands-free trivia počas šoférovania: veľké touch targety, čitateľnosť periférnym videním, voice-first) a o históriu rozhodnutí v docs/issues a git logu, ak existuje,
- poctivé priznanie, kde zdôvodnenie NEEXISTUJE (prvok je tak "lebo tak to vyšlo") — neomínaj to, nefabuluj dôvody,
- kde niečo nedáva zmysel, návrh zlepšenia s odôvodnením (návrh, nie implementácia).

Výstup: HTML artefakt docs/artifacts/ui-review-2026-07.html — po obrazovkách, dark theme, scannable, s farebným odlíšením: zdôvodnené ✅ / bez zdôvodnenia ⚠️ / navrhujem zmenu 🔧. Na záver zoznam navrhovaných zmien zoradený podľa dopadu, ako podklad na moje rozhodnutie. V odpovedi `open <path>` + krátke zhrnutie po slovensky.
```

## Session 3 — Návrhy plánovaného UI/UX na schválenie

```
Priprav vizuálne návrhy UI/UX pre plánované features/obrazovky, aby som ich mohol schváliť PRED implementáciou. Doteraz som sa rozhodoval až nad hotovou implementáciou a chcem to otočiť. Nič neimplementuj.

Vstupy: docs/artifacts/project-review-2026-07.html (sekcia issues s UI povrchom, ak existuje z predošlej session), docs/todo/TODO.md, docs/issues/, docs/product/INDEX.md, plus prípadné návrhy zmien z docs/artifacts/ui-review-2026-07.html. Z nich zostav zoznam plánovaných UI zmien a nových obrazoviek.

Použi workflow na orchestráciu (zber vstupov paralelne, návrhy mockupov po obrazovkách, krížová kontrola konzistencie štýlu); detaily orchestrácie si navrhni sám.

Pre každú plánovanú obrazovku/zmenu vytvor HTML mockup čo najbližšie finálnej podobe — rozloženie, veľkosti, texty, stavy (nie wireframe, ale ani nefunkčná appka; statický verný mockup). Drž vizuálny štýl existujúcej appky (over si ho z SwiftUI kódu / design/quiz-agent.pen cez Pencil MCP). Pri každom mockupe:

- zdôvodnenie návrhu (use-case šoférovanie, HIG, konzistencia s existujúcim UI),
- otvorené otázky/varianty, kde treba moje produktové rozhodnutie — pri variantoch ukáž obe verzie vizuálne,
- explicitnú poznámku, z ktorého issue to vychádza (číslo + názov).

Výstup: HTML artefakt docs/artifacts/planned-ui-proposals-2026-07.html — jedna sekcia na obrazovku, mockupy vložené inline, dark theme. Na záver "schvaľovací zoznam": očíslované položky, pri každej schváliť / upraviť / zamietnuť. V odpovedi `open <path>` + zhrnutie po slovensky. Implementácia čohokoľvek z toho začne až po mojom explicitnom schválení konkrétnej položky.
```

---

**Poznámky k spusteniu:** poradie 1 → 2 → 3 (session 3 využíva výstupy 1 a 2, ale vie bežať aj samostatne z backlogu). Každá session je čisto review — bez commitov do kódu; artefakty a prípadný handoff commitnúť môže.
