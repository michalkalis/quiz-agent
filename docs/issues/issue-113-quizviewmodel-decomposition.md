## Why

QuizViewModel is a 3,017-line god object across 6 files (main 1519, +Recording 697, +Audio 305, +Timers 219, +CommandListener 203, +ScenePhase 74) owning state and business logic for 8+ screens plus paywall, quota, audio devices, voice commands, and timers вЂ” no screen can be understood, tested, or changed in isolation (dim1). The `+Extension` split is file-partitioning, not decomposition: state is deliberately non-private so sibling files can reach in (~12 seam-leak markers, three with explicit "internal for вЂ¦" comments вЂ” dim6). ~36 @Published + ~17 non-observable vars express ~10 independent axes as ~50 free variables; the recording/confirmation clusters are de-facto sub-states of quizState reset by hand at 8+ sites, and `resetState` already silently misses в‰Ґ9 fields (dim3). The fix is real encapsulation: extract cohesive sub-objects with private state + explicit API that the faГ§ade composes, and fold phase-scoped fields into the phase so leaving it drops its state atomically вЂ” closing the reset-drift class of bug structurally rather than by adding more manual resets.

## Scope

**In:**
- Extract 5 sub-objects (see decision 1) with **real** encapsulation вЂ” private state, explicit narrow API вЂ” composed by a slimmed QuizViewModel acting as a thin state-machine faГ§ade that delegates, not a shared mutable-state bag.
- Close `resetState`'s в‰Ґ9 known misses via ONE unified per-child `reset()` model (decision 8, T7): each extracted child exposes `reset()` the faГ§ade `resetState`/`transition` invokes; the recording/confirmation clusters fold into `RecordingState`/`ConfirmationState` sub-structs held **private inside RecordingCoordinator** so leaving the phase drops that subset atomically; the two ownerless faГ§ade fields (activeErrorModel, mcqVoiceMatchedKey) get one explicit reset line each. Replaces the 8+ scattered manual resets вЂ” full fieldв†’mechanism map in T7.
- Derive `score`/`questionsAnswered` as computed over `currentSession` (dim3 :1259) вЂ” removes a stale-projection bug in the moved code.
- **Two dim3 cleanups ride along (recommend YES вЂ” both sit inside code being moved):** delete the dead write-only-true `autoAdvanceEnabled` axis (:148, pause is covered by currentQuestionPaused в†’ moves with QuizTimersController) and delete `CommandCapturePhase`'s unreachable `.recording/.processing` phases (`CommandCapturePhase.swift:20` в†’ the enum moves to `ViewModels/CommandCapturePhase.swift` with VoiceCommandCoordinator, T3). Both satisfy the target rule "one state machine per axis". Doing them elsewhere would re-touch the same lines.

**Out (explicit non-goals):**
- No behavior changes, no view redesigns вЂ” the decomposition is internal; user-visible flows are unchanged (behavior-pinning tests must stay green).
- Prerequisite issues, not this scope: #110 вЂ” state-machine enforcement, #111 вЂ” navigation-as-owned-state, #112 вЂ” error-path dedup. This issue inherits their fixes; it does not redo them.
- AudioService internals вЂ” #116 вЂ” AudioService split. AudioDeviceState consumes AudioService through its existing protocol facade, unchanged here.
- No `@Observable` macro migration (decision 2) вЂ” separate, larger change.

## Resolved design decisions

Each grounded in the Phase-1 research below / the review's dimensions.

1. **5 extracts, not 4.** Research correction 2: +Audio is a de-facto 5th sub-object (owns isPlayingQuestionTTS + the start/stopSilenceDetectionListening choke points + audio-device settings). Extracting only 4 leaves those straddling the boundary. **Commit to 5: EntitlementReconciler, AudioDeviceState, VoiceCommandCoordinator, QuizTimersController, RecordingCoordinator.**
2. **Composition style = faГ§ade owns children, views keep binding QuizViewModel.** Children are private/internal `@MainActor ObservableObject`s the faГ§ade holds; the faГ§ade re-publishes each child's `objectWillChange` and re-exposes each screen's slice via forwarding accessors, so view files keep `@ObservedObject var viewModel: QuizViewModel` and the test surface stays stable. Chosen over views binding children directly because that churns every view file per step and fights the one-screen-at-a-time order for no correctness gain (the encapsulation win вЂ” private child state + explicit API вЂ” holds regardless of who the view binds to); direct binding is a possible later optimization, out of scope. Repo **stays on ObservableObject** (research verdict: zero `@Observable`/`@Bindable` today; mixing = different DI/binding semantics + migration risk). Snapshot cost is unavoidable either way вЂ” the `.stableDump` recurses into child storage вЂ” so see decision 5 for handling it once.
3. **Extraction order (by self-containment): EntitlementReconciler в†’ AudioDeviceState в†’ VoiceCommandCoordinator в†’ QuizTimersController в†’ RecordingCoordinator.** EntitlementReconciler is cleanest (all methods in MAIN, only touchpoints +ScenePhase:67 and the +Recording branch #112 removes). AudioDeviceState slots 2nd вЂ” its device-management slice is the cleanest screen (AudioDevicePickerView), and pulling the shared audio primitives (TTS flag, silence choke points) out **before** their consumers lets VoiceCommand/Recording inject them per decision 4. RecordingCoordinator is most entangled в†’ last. Research's screen order (AudioDevicePicker в†’ Minimized в†’ Error в†’ Completion в†’ Result в†’ Home в†’ Question) is the per-step **verification lens**: after each object lands, exercise the screens that read its slice, Question (heaviest) verified last.
4. **Shared-state policy вЂ” no back-pointers.** `quizState`, `taskBag`, `settings`, `isAppForeground` stay resident on the faГ§ade (single source) вЂ” **and so do the three verified multi-writer cross-cluster flags: `isAutoRecording` + `isRerecording` (written by both Recording *and* Timers) and `isPlayingQuestionTTS` (written by Audio, read by CommandListener + emitEarcon).** Research ("state that resists single ownership") confirmed these have no single natural owner; they stay faГ§ade-resident and children touch them only through injected read/scoped-write closures вЂ” never by owning the field. Every other child receives the **minimal** thing it needs вЂ” a read closure, a scoped write closure (e.g. `transition(to:)`, `setAudioMode`), or a `taskBag` register/cancel handle вЂ” **never a reference to the whole VM**. Rule: *a child may hold injected values/closures, never `weak var vm: QuizViewModel`* вЂ” that is exactly what re-creates the god object through the back door.
5. **One extraction = one commit, tests ported, faГ§ade shims temporary.** Each object lands as its own commit with its mapped test file ported and green (research maps ~12 VM test files ~1:1 вЂ” EntitlementReconcileTests / QuizViewModelTimerTests / SkipCancelWord+CommandListenerTests / QuizViewModelMCQVoiceTests). FaГ§ade forwarding shims preserve the old `vm.method()`/`vm.published` API during the transition and are deleted at the end. The 4 `.stableDump` snapshot baselines ({Home,Question,Result,Paywall}) are **re-recorded once as the closing commit** вЂ” intermediate commits carry expected snapshot drift (documented in the commit, not chased); the per-step gate is the behavior suite, not the snapshot suite.
6. **Hard prerequisite: run AFTER #115 в†’ #112 в†’ #110 (in that build order).** #115 вЂ” iOS 26 raise deletes ~15 nil-branches in these very files; #112 вЂ” error dedup removes the +Recordingв†”paywall coupling (shrinks EntitlementReconciler's cross-touch); #110 вЂ” state-machine enforcement makes the transitions this faГ§ade relies on actually enforced. Each shrinks the code this issue moves. (#111 вЂ” navigation is a sibling prerequisite scope, not a build-order gate.)
7. **New files, each в‰¤ ~300 lines (repo rule).** Expected: `ViewModels/EntitlementReconciler.swift`, `AudioDeviceState.swift`, `VoiceCommandCoordinator.swift`, `QuizTimersController.swift`, `RecordingCoordinator.swift`, plus a phase-state file (e.g. `QuizState+PhaseState.swift`) for the RecordingState/ConfirmationState sub-structs. The `+Recording/+Audio/+Timers/+CommandListener/+ScenePhase` extension files empty out and are deleted as their contents move; `QuizViewModel.swift` (faГ§ade) shrinks well under 300. **Risk flag for Phase 4:** RecordingCoordinator absorbs the 697-line +Recording body вЂ” it will likely need an internal split (a confirmation sub-object or private helpers) to hold в‰¤300; that split is an impl-plan detail.

8. **Phase-state representation = `RecordingState`/`ConfirmationState` sub-structs held private inside RecordingCoordinator, NOT `QuizState` associated values.** Verified against the repo: `QuizState` (`QuizViewModel.swift:22`) is a `Sendable` enum whose recording/confirmation phases (`.recording`, `.processing`) carry **no** associated values, guarded by a custom `Equatable` that deliberately ignores associated values "to preserve animation behavior" plus a `validTransitions: Set<String>` label table. Folding phase data into associated values would force rewriting that Equatable animation contract, the transition table, and every `case .recording`/`if case` match across the codebase вЂ” high-risk plumbing churn. `RecordingState`/`ConfirmationState` **sub-structs** held as private `@Published` fields **inside `RecordingCoordinator`** (decisions 1/2 encapsulation вЂ” the same object that owns the clusters; resolves the T5/decision-8 location split), each with its own `reset()`, are the lower-risk, ObservableObject-compatible choice вЂ” a reassigned/mutated `@Published` struct fires `objectWillChange`, which the faГ§ade re-publishes per decision 2 вЂ” and touch **no** `QuizState` plumbing. Resetting is uniform (T7): every extracted child exposes `reset()` the faГ§ade `resetState`/`transition` invokes; `RecordingCoordinator.reset()` resets both sub-structs.

**Execution note:** this is the largest issue in the arch-review set вЂ” Phase 6 must take the **multi-session execution-prompts path** (`issue-113-execution-prompts.md`), one session per extract, not a single flat task list.

**No founder decision needed** вЂ” internal refactor with no product-facing surface; composition style (decision 2) is an engineering call. If Phase 4 surfaces a genuine product question, add `**Founder decision needed**` + a default and proceed.
