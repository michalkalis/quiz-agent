## Notes & deviations from spec

1. **Per the brief: "If `вЭМ REJECTED transition` is present but the app stayed alive, that's still a PASS for RS-04 (the rejection IS the guard) вАФ but note it."** In this run the guard was *more* than a transition-table rejection вАФ it absorbed the second tap **silently**, with no `REJECTED` line. Two equally plausible explanations:
   - **(a) `isStoppingRecording` short-circuit:** the second tap arrived while the first stop was still in flight; `MicButtonView` / `QuizViewModel` checked `isStoppingRecording` (or equivalent) and returned early, so no second state transition was ever attempted. This is the ideal behaviour and matches the hypothesis.
   - **(b) Mock auto-commit raced ahead of the second tap:** the mock fires `committedTranscript("Paris")` 213 ms after `startStreaming()`, which transitions `recording вЖТ processing` without the user ever tapping a stop. By the time the second tap was delivered to the SwiftUI button, the QuestionView mic button was no longer wired to a `recording`-stop action (the view re-rendered into `processing` / sheet-overlay). The tap may have hit a covered area or been routed through the now-overlaid sheet.
   - The log alone can't distinguish these вАФ both produce zero state transitions. To force the (a) path, RS-04 needs a mock variant that does *not* auto-emit on `startStreaming` (e.g. `MockElevenLabsSTTService.shouldAutoCommit = false`), so the second tap reliably lands while state == `recording`. That refinement is a candidate for a future RS-04b.
2. **`question.statusPill` not in snapshot tree** at the assertion point because the `AnswerConfirmationView` sheet overlays the QuestionView. The implicit state was read from the confirmation sheet (`confirmation.state.transcript` present, `.processing` absent вЗТ underlying state == `processing`), matching the RS-01 precedent and the spec's intent ("statusPill.AXValue вИИ {askingQuestion, recording, processing}").
3. **Step ordering for log capture (per RS-01/RS-02 lesson).** Launched with `launch_app_sim({args: ["--ui-test"]})` first, then `start_sim_log_cap({})` with no `captureConsole`. Verified `--ui-test` mode by hitting `http://127.0.0.1:9999/stt/connected` and getting 200.
4. **Tap pacing.** Both tap calls were sent in a single tool-use message (parallel). First call had `postDelay: 0`. Both XcodeBuildMCP returns came back within ~milliseconds of each other on the agent side; the underlying `xcrun simctl io вА¶ tap` events are serialised by Core Simulator but reach the app inside the 200 ms window required by the spec. The interval between the *first* tap arriving (`recording` transition at 15:24:44.061) and the second tap completing was well under 200 ms вАФ by 15:24:44.28 the mock had already auto-committed.

## Followups (separate from RS-04 verdict)

1. **Inconclusive double-tap coverage under the current mock.** As noted above, the mock's `commitAndClose`/auto-emit behaviour means the second tap may never land while state == `recording`. To prove the `isStoppingRecording` guard end-to-end, add a mock knob (`shouldAutoCommit = false` or `mockCommitDelay = 5.0`) and rerun. Filed as a candidate refinement to RS-04, not a spec bug.
2. **Same auto-confirm / `processing вЖТ processing` bug as RS-01/RS-02.** Now blocks three scenarios from cleanly exercising their intended path. Distinct issue вАФ auto-confirm timer should not call `resubmitAnswer` when the user did not edit, and `processing вЖТ processing` should be a legal no-op or the caller should be in a different state.
3. **Sentry SentryCppHelper.o ld warning** (`found architecture 'arm64', required architecture 'x86_64'`) is unrelated but noted (same as RS-01).
