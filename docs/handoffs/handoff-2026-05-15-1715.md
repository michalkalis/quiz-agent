## Decisions

- **Replayed SSE events carry `progress=0`** because the worker's `step_log` JSONB entries don't store per-step progress (only `event_id`, `step`, `started_at`, `info`). Live pubsub events carry real progress. The e2e test must therefore assert on `step` names for replay, only on `progress` for live events.
- **`Last-Event-ID` sentinel for "no prior events" is `-1`**, not `0` — because the worker's first event_id is `0` and `> last_event_id` filter would otherwise skip it on fresh connects.
- **JWS cache stores only existence**, not the parsed transaction. On cache hit, the route decodes the JWS payload locally (b64-decode middle segment) to recover `transaction_id` without re-running the full chain verify. Avoids serializing `SignedTransaction` in Redis.
- **Snapshot `.dump` is not deterministic out of the box.** The Phase 5 implementer hit three categories of noise (Combine lock pointers, `Date()` in `QuestionView`, live CFRunLoop state from `Timer.publish`) and added `.stableDump` in `HangsTests/Support/SnapshotHelpers.swift` that strips them. Future snapshot tests should use `.stableDump`, not raw `.dump`.
- **Phase 5 partial completion is acceptable.** Snapshot side green; XCUITest scaffolding compiles but cannot run until the scheme is updated. `#31` flipped to `done` with explicit note about the remaining scheme task — the alternative (leaving `[~]` indefinitely) would be worse signal.
