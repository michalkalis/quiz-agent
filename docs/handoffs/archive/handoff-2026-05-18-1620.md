## Decisions

**Lock over per-request timeout alone:** A shorter timeout without a lock still
allows N concurrent slow attempts. The lock collapses N attempts into 1 — that's
the memory-bounded property we need on a 256 MB VM.

**Code-only, no RAM bump:** User explicitly chose to keep costs at zero. 512 MB
bump is documented as the next escalation if the code guard isn't enough (see
`project_upstash_mitigation_deferred.md`).

**Deferred: lifespan also retries without fast-fail settings.** The startup
`create_pool` in `main.py:lifespan` still uses default arq retry settings
(5×, default timeout). This is acceptable because startup only runs once; the
OOM was caused by per-request stacking. If startup OOM becomes a concern, apply
`_redis_settings_fast_fail` there too.
