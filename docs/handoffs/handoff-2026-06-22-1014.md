## Decisions

- **`find_duplicates` does NOT self-exclude at the store level** — the `QuestionStore` Protocol signature is
  text-only (`find_duplicates(question_text, threshold)`), it carries no id, so the store mirrors ChromaDB and
  returns the query's own row if present. `DedupStage._is_cosine_duplicate` already filters self-matches by id
  (`match.id != question.id`), keeping re-runs idempotent. Changing the Protocol to pass an id was rejected
  (ripples to ChromaDB/Noop/DedupStage; over-engineering for one caller).
- **SYNC bridge, not a separate sync DB engine** — kept the prior session's decision. `DedupStage.run` is async
  but calls `find_duplicates` *synchronously*; `_BackgroundLoop` (`run_coroutine_threadsafe` on a daemon-thread
  loop) bridges it. Moved that pattern to `packages/shared` so both apps share one implementation.
- **Worker dedups against pgvector, not ChromaDB** — ChromaDB is frozen read-only legacy (backend.md); pgvector
  is the canonical corpus quiz-pack-api writes to. Verified safe: construction is lazy (no connect at startup),
  and the order-e2e test exercises `on_startup` + the full pipeline green.
- **DB dedup test uses a SEPARATE engine for the dedup store** — sharing the pytest `engine` fixture (bound to
  the test event loop) with a `SyncPgvectorStore` (runs queries on its background loop) throws asyncpg
  "Future attached to a different loop". Fix: seed via the fixture engine; give the dedup store its **own**
  engine over the same DB (`engine.url.render_as_string(hide_password=False)`) so each engine is used on exactly
  one loop. This mirrors production, where the worker's store is only ever reached via the bridge.
