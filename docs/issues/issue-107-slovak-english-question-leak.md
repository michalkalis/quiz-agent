# Issue 107: Slovak-language quiz occasionally serves an untranslated English question

**Triage:** bug · ready-for-agent
**Reversibility:** a
**Status:** New — from founder car-test feedback 2026-07-16; investigated via /triage 2026-07-17
**Created:** 2026-07-17

## Symptom (founder car test, 2026-07-16)

With Slovak selected as the quiz-question language, an English question was served again mid-quiz
("znova sa mi zobrazila anglicka otazka") — this is a recurrence; the founder has reported it before.
Not a full-Slovak-vs-English mode mixup: an isolated question surfaces in English while everything
else around it (UI, other questions, TTS) stays Slovak.

## Investigation — 2026-07-17

> *This was generated by AI during triage.*

**Root cause — CONFIRMED at file:line: designed silent fail-open, not a rare glitch.**

`TranslationService.translate_question` (`apps/quiz-agent/app/translation/translator.py:170-216`)
makes **one initial attempt + one retry** (`for attempt in range(2)`), then — if both attempts hit an
API exception or fail `_validate_translation` (`:103-141`, length/ratio heuristics) — falls back to
`return question` (`:216`), i.e. **the original English text**, logged only via
`logger.warning(...)` (`:212-215`). This is not a bug in the sense of an oversight: the fallback is
intentional (comment `:167-169` — "should not leak an untranslated question... mid-session"), but
the *irony* is that returning the raw original **is** the leak it says it's preventing. Nothing about
the response marks it as untranslated; the iOS client and the founder have no way to distinguish
"legitimately short Slovak answer" from "translation silently gave up."

**Why this reaches the client as the displayed AND spoken question (no divergence).** Every question
path funnels through `question_to_dict_translated` (`apps/quiz-agent/app/serializers.py:43-60`),
whose own `except` (`:58-59`) is defense-in-depth — `translate_question` itself never raises, so this
outer catch rarely fires; the `:216` in-function fallback is the one actually taken. The result is
cached once on the session (`session.current_question_text`, `flow.py:293`, `quiz.py:152`) and reused
by the audio endpoint (`tts.py:76-88`) — so a failed translation is consistently English in **both**
displayed text and TTS audio, not a text/audio split. Ruled out as a separate cause.

**Zero observability — CONFIRMED.** `main.py:50` calls `sentry_sdk.init(...)` with no
`LoggingIntegration` override, so the SDK's default `event_level=ERROR` applies; a `logger.warning`
call never becomes a Sentry event (breadcrumb only). The fallback at `translator.py:212-216` is
therefore **invisible in Sentry** — only visible in raw Fly logs, which nobody is tailing live during
a drive. This matches the founder's "I've reported this before" — there is no alerting to catch
repeat occurrences, so each one is rediscovered from scratch.

**Ruled out / not the cause:**
- **Language selection plumbing.** `session.language` is set once at session creation
  (`apps/quiz-agent/app/api/routes/sessions.py:115`) and read from the session object on every
  question/feedback/TTS call — there is no per-question client resend and no stale-language iOS
  caching path to investigate; single source of truth server-side.
- **Cache poisoning (#69 durable translation store).** Fallback/`except` returns are explicitly never
  written to the cache or the durable SQLite store (`translator.py` — store call sites are only after
  `_validate_translation` succeeds); a bad fallback does not get "stuck" and repeatedly served for
  that question text — each occurrence is a fresh live failure, not a persisted corruption. This also
  means the fallback is **retryable** (a later request for the same question can succeed) — consistent
  with the founder seeing it as intermittent rather than permanent for a given question.
- **Overlap with #100 (iOS driving-loop robustness, merged to main 2026-07-17, unshipped in build
  22).** No mention of translation/language in that issue; unrelated surface. Not fixed on main.
- **Overlap with the 2026-07-11 founder bug batch** (TODO.md) — that batch fixed TTS auth 401, dead
  voice commands, background mic, paywall reachability, tap-replay; no translation-fallback item.
  This is a distinct, previously-undocumented bug.

**Most plausible trigger for the *validation* branch (as opposed to API-error branch):** the
length-ratio guard (`translator.py:129-131`, reject if `translated/original < 0.3`) or the
short-translation guard (`:119-127`, reject if `<15` chars and original `>=30` chars) rejecting a
legitimate compact Slovak translation of a longer English question twice in a row (same prompt is
called twice with `temperature=0.3`, so both attempts can plausibly cluster around the same
length). Cannot confirm which branch (API vs validation) fired for the founder's specific occurrence
without matching Fly logs to his session timestamp — that evidence isn't available in this triage.

## Proposed fix direction

1. **Make the fallback fail loud, not silent.** When `translate_question` (or `_translate_correct_answer`
   in `flow.py:362-371`, same pattern) exhausts its attempts, emit a Sentry event (not just a log
   warning) tagged with `session_id`/`target_language`/failure reason (API error vs validation
   reject) — this alone turns "founder rediscovers it from scratch" into "we see it the moment it
   happens" and lets a future session correlate frequency instead of re-diagnosing the mechanism.
2. **Harden the retry before falling back to English:** bump attempts from 2 to 3, and/or loosen the
   validation heuristics' false-positive rate for short SK sentences (they were already relaxed once
   for T/F prompts, `:115-118` — re-check the ratio threshold against real SK/EN length data from the
   corpus).
3. Do not change the cache/store semantics (#69) — fallbacks correctly stay uncached; that's already
   the right invariant, keep it.
