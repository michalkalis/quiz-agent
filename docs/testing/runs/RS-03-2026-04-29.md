## Notes & deviations from spec

1. **The `disconnect` injection does not, on its own, route to `error`.** The
   spec assumes `curl disconnect?msg=…` mid-recording produces an error
   banner. In practice, the STT layer logs `⚠️ STT disconnected unexpectedly`
   but the QuizViewModel keeps `quizState = .recording`. The error banner only
   surfaces when the 15 s safety timer fires `commitAndClose`, the empty audio
   is submitted to the mock backend, the response has no `evaluation`, and
   `handleTranscriptionFailure()` is invoked — which transitions
   `processing → askingQuestion` and sets the banner text "Sorry, I didn't
   catch that. Please try again." That fulfils the RS-03 pre-condition (a
   pre-existing error from a failed attempt is on screen) — just via a longer
   path than the brief implies.
2. **The "flaky-mock-race" concern from the brief did not materialize.** The
   brief warned that the mock STT may auto-emit a committed transcript ~8 s
   into recording, which could race the disconnect curl. In this run the
   disconnect was injected at +3.4 s (well before any auto-commit), so the
   race did not happen. If the auto-emit timing changes in the future, fire
   the curl within ~1 s of the recording transition to stay safe.
3. **The error banner appears via the in-question overlay, not the full-screen
   error route.** RS-02 ended in the full-screen `error` route via the same
   "no evaluation" path, but only because in RS-02 the `processing → processing`
   bug fired before `handleTranscriptionFailure` could route cleanly back to
   `askingQuestion`. In this RS-03 run that bug did not fire (the timing was
   different) and the cleaner `processing → askingQuestion + errorBanner`
   path was taken — which is exactly what RS-03 needs to assert against.
4. **Mic-timer label is stale across re-records.** At the assertion snapshot
   the timer reads `● 0:08`, identical to the first recording's pre-disconnect
   value. The status-pill `AXValue` (`"recording"`) is the authoritative state
   probe; the timer text is a UI artifact and not an assertion target for RS-03.
5. **Step ordering for log capture (per RS-01/RS-02 lesson).** Launched with
   `launch_app_sim({args: ["--ui-test"]})` first, then `start_sim_log_cap({})`
   with no `captureConsole`. Verified `--ui-test` mode by hitting
   `http://127.0.0.1:9999/stt/connected` and getting 200.
6. **State name lives on `question.statusPill.AXValue`** (per RS-01/RS-02
   lesson) — the `question.state` hidden Text probe is gone (WIP in
   `QuestionView.swift`).

## Followups (separate from RS-03 verdict)

1. **Disconnect handler does not surface an error directly.** Whether this is
   correct behaviour is debatable — the user is staring at "RECORDING · SAY
   STOP TO FINISH" while the upstream is dead, and only finds out 12 s later
   when the safety timer fires. Consider whether
   `STTEvent.disconnected(error)` mid-recording should set
   `state = .error` immediately (or at least update the status pill) instead
   of silently waiting for the safety timer.
2. **Banner text is generic ("Sorry, I didn't catch that").** When the actual
   reason is `upstream-down`, surfacing that (or a "connection lost" variant)
   would help the user differentiate "speak louder" from "we have a network
   problem". Today the disconnect message is logged but never reaches the UI.
3. **Mic-timer staleness across re-records.** Cosmetic, but the timer label
   (`● 0:08`) carrying over from a prior recording into a fresh one is
   confusing — it should reset to `● 0:00` on the new recording transition.
