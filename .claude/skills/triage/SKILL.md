# /triage

Move local issues through a small state machine. This repo has **no GitHub Issues** — issues live as files in `docs/issues/issue-NN-{slug}.md` and the active queue is `docs/todo/TODO.md`.

Every comment or new section the agent appends during triage **must** start with this disclaimer:

```
> *This was generated by AI during triage.*
```

## Reference docs

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — how to write a durable agent brief that survives codebase drift
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — how the `.out-of-scope/` knowledge base works

## Where state lives

Each issue file has a header block with two structured lines:

```markdown
# Issue NN: <title>

**Triage:** <category> · <state>
**Status:** <free-text human commentary, e.g. "Fix path not yet chosen">
```

`Triage:` is the machine-readable state. `Status:` is human prose — leave it untouched unless the user asks. Older issue files only have `**Status:**` — when triaging one of those, add the `**Triage:**` line.

`docs/todo/TODO.md` is the active queue:
- `[ ]` — todo
- `[~]` — wip
- `[x]` — done

An issue can be `ready-for-agent` *and* sit in the TODO `[ ]` list. The TODO is "what's next on the runway"; triage state is "what kind of work, ready to grab or not."

## Roles

Two **category** roles:

- `bug` — something is broken
- `enhancement` — new feature or improvement

Six **state** roles:

- `needs-triage` — needs evaluation
- `needs-info` — waiting on the user / external info
- `ready-for-agent` — fully specified, ready for an autonomous (AFK) agent run
- `ready-for-human` — needs human implementation (judgment call, external access, manual testing)
- `done` — the work shipped (commit / deploy / release). The file stays as historical record.
- `wontfix` — will not be actioned

Every triaged issue should carry exactly one category and one state. If state values conflict, flag it and ask before doing anything else.

State transitions: an unlabeled / new issue normally goes to `needs-triage` first; from there it moves to `needs-info`, `ready-for-agent`, `ready-for-human`, or `wontfix`. `needs-info` returns to `needs-triage` once the user replies. The user can override at any time — flag transitions that look unusual and ask before proceeding.

## Invocation

The user invokes `/triage` and describes what they want in natural language. Interpret the request and act. Examples:

- "Show me anything that needs my attention"
- "Let's look at issue 19"
- "Move 19 to ready-for-agent"
- "What's ready for an agent to pick up?"
- "Triage the auto-confirm bug" (free-text — match against issue titles)

## Show what needs attention

Read `docs/issues/INDEX.md` first if it exists; fall back to globbing `docs/issues/issue-*-*.md` and parsing the `**Triage:**` lines. Present three buckets, oldest first:

1. **Untriaged** — files without a `**Triage:**` line, or with `needs-triage`.
2. **`needs-info`** — currently waiting on something. Surface the open questions so the user can knock them out.
3. **`ready-for-agent`** — already specified, ready to go. Useful when the user is asking "what's next."

Show counts and a one-line summary per issue. Let the user pick one.

After the bucket display, also point out any TODO `[~]` items — work-in-progress that may need state reconciliation with the underlying issue file.

## Triage a specific issue

1. **Gather context.** Read the full issue file (header, body, any prior triage notes). Parse prior triage notes so you don't re-ask resolved questions. Read `CONTEXT.md` and any ADRs in `docs/adr/` for the relevant area. Read `.out-of-scope/*.md` (lazy-create the directory) and surface any prior rejection that resembles this issue.

2. **Recommend.** Tell the user your category + state recommendation with reasoning, plus a brief codebase summary relevant to the issue (use CONTEXT.md vocabulary). Wait for direction.

3. **Reproduce (bugs only).** Before any grilling, attempt reproduction:
   - Read the reporter's steps.
   - Trace the relevant code via Read / Grep.
   - For state-machine bugs, check whether an existing RS-NN scenario in `docs/testing/regression-scenarios.md` already covers it; if it covers an adjacent scenario, run that via `/regression`.
   - Run pytest / iOS unit tests if relevant.
   Report what happened — successful repro with code path, failed repro, or insufficient detail (a strong `needs-info` signal). A confirmed repro makes a much stronger agent brief.

4. **Grill (if needed).** If the issue needs fleshing out, run a brief targeted question session (1–4 questions). Don't blow it up into a full PRD interview unless the scope warrants `/to-prd`.

5. **Apply the outcome:**
   - `ready-for-agent` — append an Agent Brief section ([AGENT-BRIEF.md](AGENT-BRIEF.md)) to the issue file. Update the `**Triage:**` line. Optionally surface to TODO.md.
   - `ready-for-human` — same brief structure, but note why it can't be delegated (judgment calls, external access, design decisions, manual testing).
   - `needs-info` — append a Triage Notes section (template below). Update `**Triage:**`.
   - `wontfix` (bug) — short polite explanation in the issue, set state to `wontfix`, mark `[x]` in TODO if listed (don't delete the issue file — keep historical record).
   - `wontfix` (enhancement) — write to `.out-of-scope/<concept>.md` ([OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)), link from a final note in the issue, set state to `wontfix`.
   - `needs-triage` — apply the role. Optional note if there's partial progress.

## Quick state override

If the user says "move 19 to ready-for-agent", trust them and apply the role directly. Confirm what you're about to do (Triage line change, optional appended brief, TODO move), then act. Skip grilling. If moving to `ready-for-agent` without a brief on the file, ask whether they want one written.

## Creating a new issue

If the user describes a new bug or feature, create `docs/issues/issue-NN-<kebab-slug>.md` where NN is the next free number (`ls docs/issues/issue-*-*.md | sort -V | tail -1` + 1). Header:

```markdown
# Issue NN: <title>

**Triage:** <category> · needs-triage
**Status:** New — created via /triage on <YYYY-MM-DD>
**Created:** <YYYY-MM-DD>
```

Then run the "Triage a specific issue" flow on it. Add a TODO line if the user wants it on the active queue.

## Needs-info template

Append to the issue file:

```markdown
## Triage Notes — <YYYY-MM-DD>

> *This was generated by AI during triage.*

**What we've established so far:**

- point 1
- point 2

**What we still need:**

- question 1
- question 2
```

Capture everything resolved during grilling under "established so far" so the work isn't lost. Questions must be specific and actionable, not "please provide more info."

## Resuming a previous session

If prior triage notes exist on the issue, read them, check whether the user has answered any outstanding questions (in the conversation, in commits, in TODO updates), and present an updated picture before continuing. Don't re-ask resolved questions.

## INDEX.md hygiene

Whenever you change a `**Triage:**` line or close an issue, update `docs/issues/INDEX.md` so it stays an accurate dashboard. If `INDEX.md` doesn't exist yet, generate it from globbing the issue files and parsing their headers. The index is the user's "what's the state of things" view — keep it current.
