# Handoff 2026-05-16 23:21

## Recap

Continued from `handoff-2026-05-16-2100.md`. Objective: unblock quiz-pack-api
deploy whose web machine was OOM-restart-looping on Upstash Redis startup
timeout from Fly CDG.

### Root cause (refined)

- **Worker** machine (also CDG) connects to Upstash successfully and stays
  connected — `redis_version=?` in arq's startup banner (Upstash free tier
  suppresses INFO fields but the connection itself is fine).
- **Web** machine, same image, same secret, same region, times out on
  `pool.ping()` during the FastAPI lifespan. arq's redis client retries 5×
  (~10s each) and then raises, which exits uvicorn. The retry storm pushes
  the 256 MB machine past the OOM limit, Fly reboots it, the cycle repeats.

`nc -zv premium-badger-116871.upstash.io 6379` from local succeeds, so the
host is reachable; the issue is intermittent CDG → Upstash global-eu2
connectivity that the worker hit at a good moment and the web kept hitting
at bad moments. **Not a code defect** — it's an infra flake the app was
crashing instead of tolerating.

### Fix shipped (commit 6ebf23e)

`fix(quiz-pack): make web startup resilient to Redis flakes`

- `apps/quiz-pack-api/app/main.py:lifespan` — wrap `create_pool(...)` in
  try/except; on failure set `app.state.arq_pool = None` and log a warning
  instead of crashing uvicorn.
- `apps/quiz-pack-api/app/api/deps.py:get_arq_pool` — if pool is `None`,
  retry `create_pool` lazily on first request; raise `HTTPException(503)` if
  still unreachable.

Effect: `/health` always answers (Fly stops OOM-looping the machine), order
endpoints degrade to 503 instead of taking the whole API down, and the queue
recovers without a redeploy once Upstash is back.
