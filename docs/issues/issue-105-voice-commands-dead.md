# Issue 105: Voice commands dead everywhere + wrong question-screen hints ("repeat"→"start")

## Symptom (founder car test, 2026-07-16)

**A — commands totally dead.** Voice commands do not work **at all, anywhere** — not in the car over Bluetooth, and not even outside the car in quiet conditions. This is on TestFlight build 22 (commit `248d63d`), which already contains #96 P2 (CmdListenBar + fail-loud asset install + Settings "Status" diagnostics row + master "Voice commands" toggle) **and** the 2026-07-11 en-US SpeechAnalyzer asset-install fix. So the earlier "assets never installed" diagnosis either regressed, never shipped correctly, or a different gate is at play.

**B — wrong recommended commands (product decision, encode it).** On the quiz/question screen the advertised commands are "repeat" and "skip"; they must be **"start" and "skip"** ("start" begins answer recording).

## Investigation — 2026-07-17

> *This was generated by AI during triage.*

**Part B — CONFIRMED, one-liner.** The per-screen hint is hard-coded at `VoiceCommandLexicon.swift:102`: question screen returns `Say "repeat" or "skip"`. The routing already supports "start" on the question screen (`QuizViewModel+CommandListener.swift:172-176`, gated on `voiceStartOnQuestionEnabled`, default `true` via `Config.voiceStartCommandEnabled` `Config.swift:156`), and the command set already includes `.start` (`VoiceCommandLexicon.swift:47`). So "start" **works today** on the question screen — only the advertised hint is wrong. Fix = change line 102 to `Say "start" or "skip"` (and drop `.repeatQuestion` from the hint; the case can stay routable).

**Part A — root cause not yet confirmed at file:line; ranked hypotheses.** These items are NOT the cause (ruled out): master toggle defaults ON (`QuizSettings.swift:95,135,174`); the listener is wired end-to-end (`QuizViewModel+Audio.swift:39,53` starts both the engine and the transcript consumer; HomeView re-arms on appear `HomeView.swift:78`).

- **H1 (PROBABLE) — speech-recognition authorization is never requested.** There is **no `SFSpeechRecognizer.requestAuthorization` / authorization check anywhere** in the iOS target (grep of `Hangs/Hangs/` finds only Apple-Sign-In `ASAuthorization` and network bearers), yet `Info.plist:69` declares `NSSpeechRecognitionUsageDescription` — i.e. the app expects to need it but never asks. The command path is the *only* consumer of on-device `SpeechTranscriber`/`SpeechAnalyzer` (`SilenceDetectionService.swift:238-245`); the answer path uses ElevenLabs STT (mic audio streamed to backend), which explains why **answers work but commands never do**. If the iOS 26 Speech API gates transcription on authorization, `analyzer.start` throws → `markCommandsUnavailable("SpeechAnalyzer start failed…")` (`:326`) → Sentry error, OR the transcriber simply never finalizes a result. Either way: total failure on every device/route, exactly as reported. (Uncertainty: whether the iOS 26 on-device `SpeechTranscriber` strictly requires the legacy authorization — verify on device / via Sentry before treating as confirmed.)
- **H2 (HYPOTHESIS) — availability never reaches `.ready` on the Slovak device.** `prepareAssets()` (`:163`) checks `SpeechTranscriber.supportedLocales`/`installedLocales` for en-US and installs on miss. If install fails or en-US isn't in `installedLocales` post-fix, availability is `.installingAssets`/`.unavailable`. Note the asymmetry: `currentCommandScreen` does **not** gate on `.ready`, so the engine still arms — but `commandListenerHint` **does** gate on `.ready` (`QuizViewModel+CommandListener.swift:70-73`), so the CmdListenBar stays hidden while no transcripts flow. The Settings "Voice commands" status row (`SettingsView.swift:664`) reads this and would show `Checking… / Installing… / Unavailable`.
- **H3 (cross-ref, not duplicate) — shared AVAudioEngine / audio-session conflict (#104 cluster).** A car/Bluetooth audio-session or two-engine conflict could kill the command engine, but that would be route-specific; the founder's failure **in quiet, outside the car** argues against this being the sole/primary cause. Track under the #104 audio cluster.

**First diagnostic step (do before coding):** read the founder's **Settings → Voice commands status row** (Ready / Installing… / Unavailable / heard "…") and run `/check-crashes` for the `.voice`-category Sentry events #96 P2 mirrors — `"Voice command assets ready|installing"`, `"Voice commands unavailable"`, `"SpeechAnalyzer start failed"`. That single reading discriminates H1 vs H2.

**Does current `main` (unshipped in build 22) already fix it?** No. #100 (merged 2026-07-17) touches the command path only for the double-submit (#100.2) and the engine-start race (`shouldStartEngine`, #100.4) — neither adds authorization nor changes availability gating. The "commands dead everywhere" symptom is untouched by #100.

## Proposed fix direction

1. **Part B (ship immediately, class a):** change the question-screen hint at `VoiceCommandLexicon.swift:102` to `Say "start" or "skip"`.
2. **Part A:** add an explicit one-time speech-recognition authorization request at launch (alongside `prepareAssets` in `AppState`), fail loud into `commandAvailability` on `denied/restricted` so the Settings row + Sentry name it. Then reconcile the availability model: either gate the actual listening on `.ready`, or make `.unavailable` reasons cover the authorization-denied case. Keep minimal — no new UI beyond the existing status row.
