## Investigation — 2026-07-17

> *This was generated by AI during triage.*

**Ruled out: the known 401 cause.** The 2026-07-11 fix "question TTS silent — attach bearer to audio downloads" (`033e6e2`) is an ancestor of `248d63d` — it **is** in build 22. That failure mode (hard 401, logged + `Self.reportAudioFailure`) is not the culprit here; also the founder reports no error banner, just silence.

**Leading hypothesis (PROBABLE): first-play races the *cold* Bluetooth HFP/A2DP negotiation, and the app double-switches the audio-session category on top of it.**

Trace of `startNewQuiz` → first TTS (`apps/ios-app/Hangs/Hangs/ViewModels/QuizViewModel.swift:588-639`):
1. `audioService.setupAudioSession(mode:)` — `AudioService.swift:212-218` — `setCategory(.playAndRecord, options: [...allowBluetoothHFP...])` then `setActive(true)`. This is the session's **first-ever activation**, which is what actually kicks off the car's Bluetooth profile handshake (HFP/A2DP). On a cold link this negotiation is known to take on the order of seconds on some head units.
2. Two awaited network round-trips follow (`createSession`, `startQuiz`, `QuizViewModel.swift:598-615`) — some but not guaranteed-sufficient settle time.
3. First `playQuestionAudio` → `playOpusAudio` → `withPlaybackCategory` (`AudioService.swift:272-292`) does a **second** `setCategory(.playback, options:[.duckOthers,.interruptSpokenAudioAndMixWithOthers])` + `setActive(true)`, moments after step 1, potentially while the car's first handshake from step 1 is still settling.

Nothing in the code waits for the route to actually include the Bluetooth port before playing (no `routeChangeNotification` await, no delay) — confirmed absent via grep across `AudioService.swift`. If CoreAudio treats the second category+activate as interrupting an in-flight route negotiation, the resulting AVPlayer can report `.playing` (KVO, `AudioService.swift:876-887`) with no exception thrown, while no audio actually reaches the car — this matches the founder's report exactly: no error banner, "just" silence. For question 2+, the same `.playAndRecord ↔ .playback` toggle (`AudioService.swift:287,292,306,309`) is flipping between two *already-negotiated* profiles on a stable link, so it's fast and reliable — consistent with "sometimes / first is worst" rather than "always."

**Same underlying mechanism as issue #104, not a duplicate.** #104 (CONFIRMED) already documents that `withPlaybackCategory` renegotiates the BT route on *every* utterance (`AudioService.swift:287`, commit `331c47c`, present in build 22) and that this flapping is real and reproducible for the mic-capture/call-mode symptoms. This issue is the TTS-output-side manifestation of the exact same flapping, with the extra first-play-specific wrinkle: question 1's category switch lands on top of a **cold** BT negotiation instead of a warm one, making silent failure most likely there. Fixing #104's flapping (hold one category for the whole quiz, or avoid the category swap entirely via `overrideOutputAudioPort`) would very likely also fix or substantially reduce this — do not build a second, separate flapping fix here.

**Not fixed on current main (unshipped).** `withPlaybackCategory` and `setupAudioSession` are unchanged between `248d63d` and `main`; #100's changes (`a669e39`, interruption `.ended` reactivation) touch a different code path (interruption recovery, not the setup→first-play window). This symptom is present in build 22 and still present on unshipped main.

**Ruled out:** `isMuted` default is `false` (`QuizSettings.swift:91,132`) with no odd decode path (`:170`) — not a mute-state bug.

## Proposed fix direction

Do not duplicate #104's fix — track the category-flapping fix there. This issue's incremental scope, once #104 lands:
- Verify whether holding one category for the whole quiz (per #104 fix direction 2) is sufficient, or whether the *first* activation additionally needs to wait for the audio route to include the Bluetooth port (or a short bounded delay) before the first `playOpusAudio` call, since #104's fix removes the *second* switch but not the inherent cold-negotiation latency of the *first* one.
- This is a timing/UX tradeoff (how long to wait, and what to do if BT never shows up) — decide with founder once #104's fix is in and this can be re-tested on-device.

## Acceptance

- [x] *(amended per #104's mode-aware fix)* The Q1 double-negotiation mechanism is gone in both modes: **Call Mode** performs zero category switches after setup (`shouldSwapCategoryForTTS` bypass, asserted in `AudioSessionCategoryOptionsTests`) — SCO opens once at setup and holds; **Media Mode** never requests HFP in either category, so its per-TTS `.playback` swap cannot renegotiate the Bluetooth profile (A2DP is the output on both sides of the swap) — the SCO↔A2DP contention this issue described cannot occur.
- [x] First-activation path inspected (2026-07-17, first-hand): `setupAudioSession` still fires-and-forgets (no route-wait) — **deliberately kept**: (a) the session first activates at app launch (`AppState`), so the car's A2DP link settles long before the first quiz TTS; (b) a genuinely stuck first play now fails LOUD via the existing 5s AVPlayer stall timer → `playbackFailed`, not silently; (c) the issue itself defers the wait-how-long/what-if-no-BT tradeoff to a founder decision after the on-device retest — unchanged.
- [ ] **[HUMAN] on-device (real car, BT), after #104's fix ships:** run several fresh quiz starts; confirm question 1's TTS is audible every time, not just questions 2+. If Q1 is still ever silent, the next lever is a bounded route-wait on first activation (founder timing/UX call).
