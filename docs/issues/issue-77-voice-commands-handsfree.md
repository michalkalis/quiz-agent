## Why

> The app used to have voice commands via the native iOS speech framework. Research the technology and the UX, analyze the app, and propose the *places* where and how voice commands should be used. My vision: keep it as simple as possible with as few commands as possible. The critical command is **start recording an answer**. **Stop recording should happen automatically** when silence is detected (nobody speaking) — beware of cabin noise in a car. The current ElevenLabs answer-recording flow is good as-is; maybe the recording UI could be polished to resemble GPT-style apps (ChatGPT, Google) — that can be a fully separate issue, agent's call. The goal is to simplify controlling the app for the driver while driving.

- **Critical command: START recording an answer.**

- Test device runs **iOS 26+**; app is used/tested in **Slovak**.
- GPT-style recording UI polish may be split into a separate issue (agent's discretion).

## Research summary (Phase 1)

Full report with citations: `docs/research/voice-commands-handsfree-research-2026-07-02.md`.

**(a) State of the code.** Spoken voice *commands* do not exist today — the `VoiceCommandService`
(SpeechTranscriber "repeat/score/help/skip" grammar) was hard-deleted in commit `c7a001c` (2026-04-21)
because "English-only voice command recognition never worked for the Slovak user." What survives is
iOS 26 `SpeechAnalyzer`/`SpeechDetector` used **purely as model-based VAD** (`SilenceDetectionService`,
1.5 s silence hangover) plus an already-built **auto-record** mode (`isAutoRecording`). The answer path is
ElevenLabs Scribe v2 Realtime streaming STT (`useElevenLabsSTT=true`, `language_code=sk`, server-side VAD
auto-commit) called directly from iOS; recording START today is a mic-button tap or auto-record after the
thinking window; STOP is automatic via VAD (ElevenLabs server VAD on streaming, Apple `SpeechDetector` on
batch/auto) with a 15 s hard cap. The `QuizState` machine is `idle→startingQuiz→askingQuestion→recording→
processing→(skipping)→showingResult→finished` (+`error`).

**(b) Prior-problems post-mortem.**
- **Root cause of removal:** the command grammar was hardcoded `en_US` and never recognised Slovak
  (`c7a001c`) — a language-coverage failure, not a crash. iOS 26 confirms this is structural: on-device
  `SpeechTranscriber.supportedLocales` has **no Slovak**.
- **Self-trigger:** a broken thinking-timer once let recording start *during* TTS playback (`1a19438`, fixed).

  detector↔transcriber pairing) — proof SpeechAnalyzer breaks under iOS point releases; design defensively.
- **Audio-session conflicts:** #59.3 (missing `.allowBluetoothHFP` → AirPods mic dead) fixed; **#67 still
  OPEN** — the interruption handler misses the streaming path, and barge-in is architecturally dead
  (`isTTSPlaybackActive` never set true; VAD torn down before TTS).
- **False-activation:** #66 (backend ghost-question / silent session-advance on unrecognised intent) still
  open — a real risk for any new command grammar in a noisy car.
- **CI blind spot:** the suite mocks `AVAudioSession`, so real audio/BT regressions are invisible (RS-01..10

## Research delta (2026-07-03)

Full report: `docs/research/voice-commands-native-english-delta-2026-07-03.md`. Reworks only what the founder
feedback changed; the 2026-07-02 report stays valid for audio topology, crash history, and STOP tuning.

- **The deleted `VoiceCommandService` (`c7a001c`) used exactly what we're re-adopting:** `SpeechAnalyzer` +
  `SpeechTranscriber` hardcoded `en_US`, matched by **exact word-boundary** (13-word enum), **no fuzzy/phonetic
  layer, no contextual biasing** (options were empty), continuous listen. It failed because accented English
  mis-transcribed and an exact match turned a near-miss into a no-match. Lessons: keep word-boundary
  tokenization but **add per-token phonetic/edit-distance tolerance** + confidence floor + screen scoping;
  keep buttons as the additive fallback.
- **No custom-vocabulary/`contextualStrings` on the new framework** — biasing is unavailable on the locked-in
  native path (a real loss vs legacy `SFSpeechRecognizer`); accent-robust word choice + fuzzy matching is the
  whole mitigation. `DictationTranscriber` (short-utterance module) may beat `SpeechTranscriber` for a tiny
  command grammar — A/B on device.
- **Cheapest integration:** `SilenceDetectionService` already instantiates an unused paired `SpeechTranscriber`
  (`:106-113`) — re-locale it to English and consume its `.results`: no new engine, same tap. Because START is
  now button/timer, the command listener and the ElevenLabs answer stream are **time-disjoint**, so the 77.7
  single-engine pin still holds and simplifies.
- **Windowed, not always-on:** listener armed only on Home/Question/Confirmation/Result, torn down during TTS
  (self-trigger guard) — no AEC needed. iOS 26.3 `start(inputSequence:)` streaming regression + Simulator
  `supportedLocales`-empty ⇒ defensive fallback to button-only + a `[HUMAN]` on-device accent gate are mandatory.
- **Recommended command set (accent-chosen, English):** `start` (open mic on Question — recovers hands-free
  START without auto-arm), `ok`, `next`, `repeat`, `skip` (+ optional `stop`). STOP-on-silence stays automatic.
- **Pencil (`design/quiz-agent.pen`) implications** (not edited here): listening indicator + per-screen command
  hints on Home/Question/Confirmation/Result; reworked onboarding voice explainer (English-only); every voice
  affordance keeps a visible button twin.
- **Open product Qs for Phase 2:** (1) confirm spoken "start" opens the mic on Question (vs button-only START);
  (2) lock the final word set (provisional until the on-device accent test).

## Scope

**In scope** — the hands-free voice-control *mechanism*: a minimal **native-English spoken-command** layer (start · repeat · skip · ok · next), robust automatic STOP, eyes-free confirmation. START stays button/timer with a spoken "start" as the hands-free recovery.

- **START stays the existing button/timer flow — no auto-mic-open (founder-locked, decision 1).** After the question TTS finishes the app does **not** open the mic; the existing thinking-timer + mic-button flow is unchanged. The dropped cycle-2 VAD auto-arm + pre-roll are OUT. Hands-free START is recovered instead by a spoken **"start"** command on `QuestionView` (coordinator-adopted default — see provenance; founder-overridable to button-only START, which drops only the "start"-opens-mic wiring, not the rest of the command layer).
- **Native-English spoken commands via the iOS speech framework (founder-locked, decision 2).** Commands are recognised **on-device in English by Apple's `SpeechAnalyzer`** (`DictationTranscriber` preferred for a tiny command grammar, `SpeechTranscriber` fallback — A/B on device per delta §B1), **English-only for every user regardless of app language.** This is a separate recognizer from the Slovak ElevenLabs answer path — **not** transcript-matching on the answer stream (that design is reversed). No new third-party ASR, no wake-word.
  - **Cheapest integration — reuse the already-paired, currently-unused transcriber (delta §A2/L5).** `SilenceDetectionService` already instantiates a `SpeechTranscriber` alongside its `SpeechDetector` (to satisfy the CARQUIZ-3 forced detector↔transcriber pairing) whose `.results` stream is **never consumed** (`SilenceDetectionService.swift:106-113`; only `detector.results` is read at `:195`). Re-locale that transcriber to **English** and add one async consumer loop → English command listening with **no new engine, same tap, same buffers.** Lowest-risk seam.
  - **Windowed, per-screen listener — never always-on (delta §A2, §B1).** The command listener is armed only on the screens/states below and **torn down during TTS** (self-trigger guard — an always-on English listener would hear the app's own English TTS; the `1a19438` lesson). Windowing also avoids battery drain and the documented iOS 26.3 long-lived-streaming `start(inputSequence:)` regression.
  - **Screen / command map (delta §A3):**
    - **HomeView (`idle`):** "start" → begin the quiz.
    - **QuestionView (`askingQuestion`, after TTS):** "start" → open the mic / begin recording an answer (the hands-free START recovery); "repeat" → replay the question audio (wire the currently-dead `repeatQuestion()`, `QuizViewModel.swift:1017`) and re-arm; "skip" → skip, behind the skip guard below.
    - **QuestionView (`recording`):** *no command listening* — the answer window is 100% ElevenLabs capture and STOP is automatic on silence. Adding command-listening here would fight the answer stream and reintroduce the concurrency this design avoids.
    - **AnswerConfirmationView (sheet, `processing`):** "ok" → confirm; "again"/"retry" → re-record. The existing **10 s auto-confirm stays the no-speech default**; buttons stay as fallback.
    - **ResultView (`showingResult`):** "next" (or "ok") → advance. Auto-advance unchanged.
    - **CompletionView / Settings / Paywall:** no voice.
  - **Matching approach (the v1 failure lesson — delta §A1/L1-L2, §B2).** The original `VoiceCommandService` (`c7a001c`) failed because a Slovak-accented English speaker mis-transcribed ("start"→"stat") and its **exact word-boundary match** turned every near-miss into a no-match. The new matcher keeps word-boundary tokenization (kills the substring false-positive class) but **adds per-token phonetic / edit-distance tolerance + a confidence floor**, **screen-scoped** so each token competes against only the 1–2 valid commands on that screen (wide fuzzy margins) — a sibling of the shipped `MCQTranscriptMatcher`. **Honest limitation:** the new `SpeechAnalyzer` framework exposes **no `contextualStrings`/custom-vocabulary biasing** (unlike legacy `SFSpeechRecognizer`), so we cannot bias the model toward the command words — accent-robust word choice + fuzzy matching is the *entire* mitigation (delta §B1).
  - **Skip protection (founder-mandated: "skip carefully").** Two layers: **(a) strict whole-utterance match** — the transcript must *be* the skip word (modulo filler), not merely contain it; then **(b) a skip-confirm earcon + a short (~2.5 s) undo window** during which a spoken cancel word ("stop"/"no") or a tap aborts before skip commits. Everything else stays simple; skip is the one place a false positive is costly.
  - **English command lexicon (accent-chosen — delta §B2).** Short, stressed, consonant-anchored, mutually-distant words that survive a Slovak accent and exist as Slovak loanwords: **start, ok, next, repeat, skip** (+ optional **stop** to cancel — provisional). Screen-scoped so context disambiguates ("ok" = confirm on the sheet, = advance on the result). Lives in a constants file next to `MCQTranscriptMatcher`. **This word set is provisional pending the on-device Slovak-accent test — final tuning happens at the `[HUMAN]` gate; be prepared to swap words** (delta §B2, §C risk 1).
- **Single shared `AVAudioEngine` + one input tap (architecture pin — 77.7, re-validated for the native topology — delta §A2, §D).** Today two engines run on the same mic — the VAD path (`SilenceDetectionService.swift:125`, tap `:162`) and the ElevenLabs streaming path (`AudioService.swift:609`, tap `:652`) — and they run **concurrently** during an ElevenLabs answer (Engine A re-armed after TTS at `QuizViewModel+Audio.swift:95`, never torn down before Engine B spins up). That is the documented "two concurrent `AVAudioEngine`s" config behind the `AVAudioEngine`+`AVPlayer` SpeechAnalyzer crash / open concern #64. **Pin: one shared engine + one tap fanning PCM to (a) `SpeechDetector` VAD, (b) ElevenLabs `sendAudioChunk` (answer stream), and (c) the new English command `SpeechTranscriber`.** Because START is button/timer, the command listener (c) and the answer stream (b) are **time-disjoint — never live at the same instant** — so the pin holds and is *simpler* than the old concurrent design. `startStreamingRecording` must consume the shared tap, never instantiate a second engine (`ElevenLabsSTTService` owns no audio hardware — it only consumes PCM via `sendAudioChunk`).
- **Tune the existing stop-on-silence for the car cabin.** STOP stays automatic on the two shipping model-based VADs (Apple `SpeechDetector` on-device; ElevenLabs server-side on the streaming path). Tune the silence hangover, add a min-speech-duration, and lower `SpeechDetector` sensitivity for road noise (starting points in the decisions table). No new VAD engine — no Silero, no energy/RMS threshold.
- **Earcons for eyes-free confirmation (minimal set — language-neutral).** A distinct "mic is live" tone when the mic opens, a "got it" tone at STOP, a **skip-confirm** tone for the skip undo window, and a light **command-ack** cue when a spoken command is recognised — so the driver never looks down. Language-neutral and load-bearing for eyes-free use. Absorbs #68's record-start/stop earcon item (see below).
- **The screen / state map.** Command listening applies on `HomeView` (`idle`), `QuestionView` (`askingQuestion` only — armed after TTS, torn down during recording), `AnswerConfirmationView` (`processing`) on top of the 10 s auto-confirm, and `ResultView` (`showingResult`). **Glanceable buttons remain the fallback on every screen — voice is additive, never the only path.** No voice during the recording window, none on Settings/Paywall. A single observable **capture-phase** (`idle → armed → listening → recording → processing`) is the source of truth for earcons and the deferred UI (see D11 subsection).
- **Robustness guards against the known failure modes:**
  - *Self-trigger during TTS* — the command listener (like VAD) is fully torn down during TTS and armed only after TTS completes; windowing removes self-trigger without AEC (the `1a19438` lesson; delta §A2).
  - *Accent brittleness + command mis-fire → #66 interplay* — fuzzy/phonetic matching + confidence floor + screen scoping absorb accent mis-transcription; the strict-skip guard + undo window protect the one costly action. Because the command listener does **not** run during the answer window, a mis-recognised command cannot become an answer submission there — the command layer is architecturally isolated from the answer path (a real simplification over the cycle-2 transcript-matching design). The residual #66 exposure — a genuine non-answer intent still reaching the backend via the answer submit — is caught by #66's early-return guard (see Dependencies).
  - *SpeechAnalyzer fragility across iOS point releases* — a defensive wrapper so that if transcriber/detector setup fails or throws (CARQUIZ-3-class drift, the 26.3 `start(inputSequence:)` regression, or `supportedLocales` empty) the flow degrades to the existing manual mic-button / tap flow instead of crashing (< iOS 26 already no-ops to button control). Mandatory, not optional (delta §B3, §C risk 3).
- **Dependencies (locked ordering).**
  - **#66 (ghost-question early-return) — HARD PREREQUISITE; lands before #77's command layer.** Any non-answer intent that reaches `POST /api/v1/sessions/{id}/input` (e.g. a genuine mis-parse of a spoken answer) hits the exact submit path #66 mishandles; without #66's `if result.evaluation is None: return` guard (`flow.py:256-302`) it silently advances the session and burns a freemium question. Carried verbatim from cycle-2 — the required backstop.
  - **#67 Part A (interruption handler misses the streaming path) — lands before or within #77.** #77's confirmation-sheet listening window and any streaming changes keep the stranded-recording-after-a-call bug (`AudioService.swift:385-388` calls the batch `stopRecording()`, never `stopStreamingRecording()` `:704`) in scope: **Part A ships first, or in the same change set, and #77 must not regress it.** #67 Part B (barge-in) stays OPEN and out of scope.
- **Pencil design update (IN SCOPE — decision 6; delta §A4).** Update `design/quiz-agent.pen` **via the Pencil MCP tools** (the executor must use pencil MCP — `.pen` files are encrypted): add a **listening indicator** (distinct from the answer-recording LISTENING card), **per-screen command hints** ("Say 'start'", "Say 'ok' / 'again'", "Say 'next'") on Home / Question / Confirmation / Result, and rework the **onboarding voice explainer** to teach the small English command set and note it is **English-only by design**. Every voice affordance keeps a **visible button twin** — the design must not imply voice-only control.
- **Verification strategy (honest sim-vs-device split).**
  - **Machine-checkable headless (unit + sim — this is what gates the loop):** capture-phase state-machine tests driven by *injected* events; command-matcher unit tests over recorded/synthetic **English** transcripts (start/repeat/skip/ok/next routing, strict-skip whole-utterance guard, screen-scoped disambiguation, undo-window timer, non-command → no match); and the confirmation-sheet Confirm/Re-record window logic, with the Apple recognizer mocked.
  - **`[HUMAN]` on-device pass (cannot be headless):** the real Apple **`SpeechTranscriber`/`DictationTranscriber` arm → English recognition** step (`supportedLocales` returns **empty on the Simulator**; transcriber/detector run only on device) — including the **founder's own Slovak-accented English** as the real acceptance test, final word-set tuning, cabin-noise VAD tuning, Bluetooth HFP/A2DP routing, and phone-call interruption recovery. **Net:** command-routing logic, the state machine, and confirmation logic reach a machine-verified done-state headless; the on-device English recognition, accent robustness, and noise/BT/interruption behaviour carry an explicit `[HUMAN]` gate — #77 is **not** fully headless-closable, stated rather than hidden.

### How #66 / #67 / #68 are affected

- **#66 (backend ghost-question) — HARD PREREQUISITE.** A non-answer intent that reaches the answer submit path (a genuine mis-parse of a spoken answer) travels the exact path #66 mishandles. Unlike cycle-2, the command listener does **not** run during the answer window, so a mis-recognised *command* cannot become an answer there — the exposure is narrower — but #66's early-return guard (`flow.py:256-302`, `if result.evaluation is None: return`) is still the **required backstop and must land before #77's command layer.** It remains #66's own change (backend), not #77's.
- **#67 (audio interruption & barge-in).** **Part B (barge-in) stays OPEN** — decision 3 does not build AEC; #77 relies on strict sequencing and does **not** rip out or wire the dead barge-in infra (that call stays with #67). **Part A (interruption handler misses the streaming path) — hard ordering: lands before or within #77.** The confirmation-sheet listening window and streaming changes keep the stranded-recording-after-a-call bug (`AudioService.swift:385-388` batch `stopRecording()` never calling `stopStreamingRecording()` `:704`) in scope. Part A ships first, or in the same change set; #77 must not regress it.
- **#68 (driving UX defaults & earcon).** #77 **absorbs #68's record-start/stop earcon item** — the minimal earcon set is inseparable from safe eyes-free command control and ships here (language-neutral and still load-bearing); #68 should mark that item delivered-by-#77. #68's **thinking-time default (60 s → 10 s) + Settings exposure** and **image-question rendering** stay in #68.

### Second-order lens (D11) — system & roadmap effects

- **Multiplayer (post-MVP).** The spoken commands are speaker-agnostic — they act on whoever speaks first, fine for single-participant MVP but needing a "whose turn" gate later. The capture-phase observable, the windowed-listener seam, and the single command matcher leave room to add participant-scoping without reworking the command layer; nothing here forecloses it.
- **Freemium daily-limit flow.** A mis-recognised command must never silently burn a daily-limited question. Two structural protections: (a) the command listener is **off during the answer window**, so a command can never be mis-submitted *as* an answer there; and (b) the #66 backend early-return catches any genuine non-answer intent that still reaches the submit path. This is why #66 is a hard prerequisite here, not an afterthought.
- **iOS point-release fragility.** SpeechAnalyzer has broken twice under point releases (CARQUIZ-1 Swift-6 tap; CARQUIZ-3 iOS 26.3 detector-pairing) plus the 26.3 `start(inputSequence:)` streaming regression. The mandatory defensive wrapper degrades a transcriber/detector failure to manual button/tap control instead of crashing — a platform regression becomes a graceful downgrade, matching the existing `< iOS 26 → nil service → button-start` pattern. Delta §B3.
- **Deferred GPT-style UI.** The capture-phase enum (`armed / listening / recording / processing`) is defined now precisely so the later UI-polish issue can bind an orb/waveform to real states without re-plumbing the view-model — the state that UI will need is exposed at the seam today.
