## Scope

**In scope** — the hands-free voice-control *mechanism*: robust START, a minimal **spoken-command** layer (Repeat / Skip · Confirm / Re-record), robust STOP, eyes-free confirmation.

- **VAD-triggered auto-start with pre-roll.** After the question TTS finishes and a "mic live" earcon plays, the mic arms and the *first detected speech* starts capture; a rolling pre-roll buffer (~300 ms) is prepended so the first syllable isn't clipped. Reuses the on-device `SpeechDetector` already in `SilenceDetectionService`. No spoken keyword. *(This is the coordinator-adopted default START — see decision 1 provenance. The founder may override to button-only START before execution; that override drops only this auto-arm glue, not the spoken-command layer below.)*
- **Spoken commands, recognized from the existing ElevenLabs transcript (NEW — reverses the cycle-1 "buttons only" call).** The answer path already streams Slovak STT (ElevenLabs Scribe v2 Realtime, `language_code=sk`). Commands are natural Slovak words matched **client-side on the committed transcript** — **no new ASR engine, no on-device Slovak recognizer, no wake-word.**
  - **During the answer window** (mic open on `QuestionView`): the committed transcript is classified *before* it is submitted as an answer. **"zopakuj (otázku)"** → replay the question audio and re-arm (a pure client action — no backend). **"preskoč / ďalšia (otázka)"** → skip, behind the skip guard below. Anything that is not a command falls through to the existing answer path.
    - **Where matching happens = client-side, at `handleCommittedTranscript` (`QuizViewModel+Recording.swift:173`), inserted ahead of the existing answer-submission branch** — the same seam and pattern as the shipped `MCQTranscriptMatcher` (`:204`, diacritic-fold + tiered match). Rationale: (1) **Repeat has no backend action** (it replays already-downloaded audio) so it *must* be client-side; (2) it keeps the LLM/network **off the answer hot path** — matching is an instant local string test, no round-trip, which matters for driving latency; (3) it **intercepts a command before it can reach the backend as a mis-parsed answer**, which is exactly the #66 ghost-question path (a non-answer intent that today advances the session and burns a daily-limit question). The alternative — POST everything to the backend and let intent classification decide — routes every "zopakuj" straight into the #66 bug and adds a round-trip; rejected. All voice-answer paths converge on `NetworkService.submitTextInput` → `POST /api/v1/sessions/{id}/input`, so a single client-side gate ahead of that call covers the whole surface.
    - **Skip protection (founder-mandated: "skip carefully — the user must not accidentally skip a question").** Two-layer guard: **(a) strict whole-utterance match** — the transcript must *be* a skip phrase (modulo filler), not merely *contain* "preskoč", so an answer like "to sa nedá preskočiť" is not read as a skip; then **(b) a skip-confirm earcon + a short (~2.5 s) undo window** during which a spoken **"späť / nie"** or a tap cancels before the skip commits to the backend. Repeat needs no such guard (idempotent, harmless). Skip is the one place a false positive is costly, so it gets the strict-match-plus-undo belt-and-suspenders; everything else stays simple.
  - **Answer-confirmation sheet — a short listening window (NEW).** When `AnswerConfirmationView` appears, open a brief listening window on the **same ElevenLabs streaming path** matching **"potvrď / pošli / áno"** → confirm and **"znova / ešte raz"** → re-record. The existing **10 s auto-confirm remains the no-speech default**; buttons stay as fallback. Cost lens: a few extra seconds of ElevenLabs Scribe v2 Realtime streaming per question during the window — a small, honest add to streaming minutes on an engine already in use.
  - **Slovak command lexicon (design decision).** Short, unambiguous, common words; **screen-scoped so on-screen context disambiguates overlaps** (e.g. "ešte raz" = re-record on the sheet, never repeat during the answer window): Repeat = {zopakuj, zopakuj otázku, opakuj}; Skip = {preskoč, preskočiť, ďalšia, ďalšia otázka}; Confirm = {potvrď, pošli, áno, ok}; Re-record = {znova, ešte raz, nahrať znova}; Undo/cancel = {späť, nie, zruš}. Lives in a constants file next to `MCQTranscriptMatcher` (Slovak now, tracking the STT content-language; localizable when other STT languages are added — a recognition lexicon, not a UI-string catalog).
- **Single shared `AVAudioEngine` for the arm→stream handoff (architecture pin — resolves the pre-roll-engine-handoff caution).** Today there are **two independent engines** running on the same mic: the VAD path (`SilenceDetectionService.swift:125`, tap `:162`) and the ElevenLabs streaming path (`AudioService.swift:609`, tap `:652`) — and they already run **concurrently** during an ElevenLabs answer today (Engine A is re-armed after TTS at `QuizViewModel+Audio.swift:95` and never torn down before Engine B spins up in `startStreamingRecording`). That is already the documented "two concurrent `AVAudioEngine`s" configuration behind the `AVAudioEngine`+`AVPlayer` SpeechAnalyzer crash and open concern #64 — VAD-triggered START only makes the handoff more load-bearing. **Pin: one shared engine + one input tap; the single tap fans PCM out to both (a) the `SpeechDetector` analyzer input and (b) a ~300 ms pre-roll ring buffer.** On `speechStarted` the ElevenLabs websocket (`ElevenLabsSTTService`, which owns no audio hardware — it only consumes PCM via `sendAudioChunk`) starts draining the pre-roll ring buffer, then live frames, from that **same** tap — so streaming START **never instantiates a second engine.** Converging the two engines onto one tap is the load-bearing build item here (not merely "add pre-roll"); it also removes today's latent running-but-unused Engine-A-during-streaming crash surface.
- **Tune the existing stop-on-silence for the car cabin.** Adopt + tune the two model-based VADs already shipping (Apple `SpeechDetector` on-device; ElevenLabs server-side on the streaming path). Tune the silence hangover, add a min-speech-duration, add prefix-padding, and lower `SpeechDetector` sensitivity for road noise (starting points in the decisions table). No new VAD engine — no Silero, no energy/RMS threshold.
- **Earcons for eyes-free confirmation** — a distinct "mic is live" tone at START and a "got it" tone at STOP, plus a **skip-confirm earcon** for the skip undo window and a light **command-ack** cue when a spoken command is recognized, so the driver never looks down. This is the layer that makes VAD-start and spoken commands safe. Absorbs #68's record-start/stop earcon item (see below).
- **The screen / state map.** VAD-start + Repeat/Skip commands apply on `QuestionView` (`askingQuestion → recording`). `AnswerConfirmationView` (`processing`) gains the short Confirm/Re-record listening window on top of the existing 10 s auto-confirm. `ResultView` (`showingResult`) auto-advances. **Glanceable Confirm / Re-record / Repeat / Skip buttons remain the fallback on every screen — voice is additive, never the only path.** No voice on Home / Settings. A single observable **capture-phase** (`armed → listening → recording → processing`) is the source of truth for both the earcons and the future UI (see D11 subsection).
- **Robustness guards against the known failure modes:**
  - *Self-trigger during TTS* — strict sequencing: VAD/mic stay fully torn down during TTS and arm only after TTS completion + earcon (the `1a19438` lesson).
  - *False activation in noise + command mis-fire → #66 interplay* — min-speech-duration rejects a cough / road-noise blip as a false start; and because commands are intercepted **client-side before submission**, a recognized command never reaches the backend as an answer. But client-side matching is not perfect: any command utterance the matcher **misses** is submitted as an answer and hits the backend intent classifier — precisely the #66 non-answer-intent path that burns a daily-limit question and desyncs the session. **#66's early-return guard is therefore a hard prerequisite (see Dependencies)** — the backstop for every missed match; the strict-skip guard and min-speech-duration reduce, but do not remove, that exposure.
  - *SpeechAnalyzer fragility across iOS point releases* — a defensive wrapper so that if detector setup fails or throws (CARQUIZ-3-class drift), the flow degrades to the existing manual mic-button START instead of crashing (< iOS 26 already no-ops to button-start).
- **Dependencies (locked ordering).**
  - **#66 (ghost-question early-return) — HARD PREREQUISITE; lands before #77's command layer.** Reversing to spoken commands routes non-answer utterances through the exact submit path #66 mishandles; without #66's `if result.evaluation is None: return` guard (`flow.py:256-302`), a single missed client-side match silently advances the session and burns a freemium question. This is **no longer "out of scope / its own issue"** as cycle-1 framed it — #77's spoken-command acceptance depends on it. The client-side interception is the primary defense; #66 is the required backstop.
  - **#67 Part A (interruption handler misses the streaming path) — lands before or within #77.** #77's auto-arm and the new confirmation-sheet listening window both **increase time-in-streaming**, so the stranded-recording-after-a-call bug (`AudioService.swift:385-388` calls the batch `stopRecording()`, never `stopStreamingRecording()` `:704`) becomes more likely to be hit. Locked as a hard ordering line, not a soft note: **Part A ships first, or in the same change set, and #77 must not regress it.** #67 Part B (barge-in) stays OPEN and out of scope (decision 2).
- **Verification strategy (honest sim-vs-device split — replaces the cycle-1 "one real-audio smoke test" line, which over-claimed a headless close of the CI blind spot).**
  - **Machine-checkable headless (unit + sim — this is what gates the loop):** capture-phase state-machine tests driven by *injected* `SilenceEvent`s (`.speechStarted` / `.silenceAfterSpeech`); command-matcher unit tests over recorded/synthetic Slovak transcripts (repeat vs skip vs answer routing, strict-skip whole-utterance guard, undo-window timer); and — because the ElevenLabs path is **network STT that runs on the Simulator** — an **end-to-end sim test of command routing** (audio → ElevenLabs transcript → matcher → command vs submit) covering the answer-window commands and the confirmation-sheet Confirm/Re-record window, with the Apple detector mocked.
  - **`[HUMAN]` on-device pass (cannot be headless):** the real Apple **`SpeechDetector` arm → speechStarted → silence-stop** step — `SpeechTranscriber.supportedLocales` returns **empty on the Simulator** and `SpeechDetector` runs only on device — plus cabin-noise tuning (hangover / sensitivity / min-speech-duration), Bluetooth HFP/A2DP routing, and phone-call interruption recovery. **Net:** #77 reaches a machine-verified done-state for command routing, the state machine, and confirmation logic headless; the Apple-VAD START trigger and the noise/BT/interruption robustness carry an explicit `[HUMAN]` on-device gate — #77 is **not** fully headless-closable, and this is stated rather than hidden.
- **Doc hygiene** — correct the now-false "voice commands always available" text the research flagged (`CONTEXT.md` Repeat/Mute/Skip glossary entry, `OnboardingView` "Say 'skip'…", `Logging.swift` `Logger.voice` comment) and the dead `repeatQuestion()` reference, so the docs describe the **new voice-plus-button reality** (Repeat/Skip and Confirm/Re-record are spoken *or* tapped).

**Out of scope:**

- **Barge-in / AEC** — no `.playAndRecord` + `.voiceChat`, no `setPrefersEchoCancelledInput`, no audio-session redesign to talk over the question. Strict TTS→mic sequencing instead. Leaves #67 Part B open. *(founder-confirmed, decision 2)*
- **A new ASR engine or any on-device Slovak recognizer** — commands are matched from the *existing* ElevenLabs transcript, not a rebuilt grammar; no Porcupine, no Slovak on-device model, no always-on wake-word. *(This narrows — not deletes — the cycle-1 "no spoken command grammar" line: spoken commands are now IN scope, but only via transcript matching.)*
- **GPT-style recording-UI polish** — animated orb/waveform listening/thinking/speaking states → a separate follow-up issue (not created by this phase; pointer only).

### How #66 / #67 / #68 are affected

- **#66 (backend ghost-question) — HARD PREREQUISITE (upgraded from cycle-1 "out of scope").** Reversing to spoken commands means non-answer utterances ("zopakuj"/"preskoč") travel the exact submit path #66 mishandles. #77's primary defense is **client-side interception** (a recognized command never reaches the backend as an answer) plus a min-speech-duration guard, but any *missed* client-side match is POSTed as an answer and hits the #66 non-answer-intent path — silently advancing the session and burning a freemium daily-limited question. So #66's early-return guard (`flow.py:256-302`, `if result.evaluation is None: return`) is the **required backstop and must land before #77's command layer.** It remains #66's own change (backend), not #77's — but the ordering is now locked, not optional.
- **#67 (audio interruption & barge-in).** **Part B (barge-in) stays OPEN** — decision 2 does not build AEC; #77 relies on strict sequencing and does **not** rip out or wire the dead barge-in infra (that call stays with #67). **Part A (interruption handler misses the streaming path) — hard ordering: lands before or within #77.** #77's auto-arm and the new confirmation-sheet listening window both increase time-in-streaming, making the stranded-recording-after-a-call bug (`AudioService.swift:385-388` batch `stopRecording()` never calling `stopStreamingRecording()` `:704`) more likely to be hit. Part A ships first, or in the same change set; #77 must not regress it. (Upgraded from cycle-1's soft "should ideally land before/with.")
- **#68 (driving UX defaults & earcon).** #77 **absorbs #68's record-start/stop earcon item** — the earcon is inseparable from safe eyes-free VAD-start and ships here; #68 should mark that item delivered-by-#77. #68's **thinking-time default (60 s → 10 s) + Settings exposure** and **image-question rendering** stay in #68; #77 only *depends on* a short arm-delay (its arm-then-wait-on-VAD design must not rely on a 60 s timer).

### Second-order lens (D11) — system & roadmap effects

- **Multiplayer (post-MVP).** VAD-triggered start *and* the spoken commands are speaker-agnostic — they act on whoever speaks first, fine for single-participant MVP but needing a "whose turn" gate later. The capture-phase observable, the "arm after earcon" seam, and the client-side command matcher (a single interception point) leave room to add participant-scoping without reworking START or the command layer; nothing here forecloses it.
- **Freemium daily-limit flow.** Neither auto-start on noise **nor a mis-recognized spoken command** may silently burn a daily-limited question. Two load-bearing protections now stack: (a) min-speech-duration + "no speech → cancel, don't submit" on START, and (b) client-side command interception ahead of the submit call so "zopakuj"/"preskoč" never POST as an answer. Both *reduce* the exposure; the **residual** — a command the client matcher misses — is caught only by #66's backend early-return, which is why #66 is a hard prerequisite here, not an afterthought.
- **iOS point-release fragility.** SpeechAnalyzer has broken twice under point releases (CARQUIZ-1 Swift-6 tap; CARQUIZ-3 iOS 26.3 detector-pairing). The defensive wrapper degrades a detector failure to manual button-START instead of crashing — a platform regression becomes a graceful downgrade, matching the existing `< iOS 26 → nil service → button-start` pattern.
- **Deferred GPT-style UI.** The capture-phase enum (`armed / listening / recording / processing`) is defined now precisely so the later UI-polish issue can bind an orb/waveform to real states without re-plumbing the view-model — the state that UI will need is exposed at the seam today.
