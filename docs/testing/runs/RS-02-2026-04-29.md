## Notes & deviations from spec

1. **Hypothesis vs. reality of the mock.** RS-02's hypothesis is "When no committed transcript ever lands, the 15 s safety timer must stop recording". In practice, when the auto-stop timer fires, `QuizViewModel.stopRecordingAndSubmit()` calls `sttService.commitAndClose()`. The DEBUG-only `MockElevenLabsSTTService.commitAndClose()` *yields* `STTEvent.committedTranscript("Paris")` (default `mockCommittedText`). So a committed transcript *does* land ŌĆö but only as a side-effect of the auto-stop path, not from any external injection. The end-state assertion (`state ŌēĀ recording`) still holds, but the scenario as written doesn't strictly exercise the "no transcript ever arrives" branch under the current mock. To exercise that branch, the mock would need a `shouldEmitOnCommit = false` flag or the mock's `commitAndClose` would need to no-op.
2. **The `error` state is reached via a known-buggy path, not a clean timeout error.** The `processing ŌåÆ error` transition at 15:15:33 is the same followup bug noted in `RS-01-2026-04-29.md` ŌĆö auto-confirm timer fires `resubmitAnswer` ~10 s after entering `processing`, the legal-transition table rejects `processing ŌåÆ processing`, then the mock's `previewStartQuiz` response has no `evaluation`, so we fall into `setError("Could not evaluate your answer. Please try again.")`. RS-02 assertions still pass because the resulting error string is human-readable and the app is alive ŌĆö but this is incidental, not the documented behavior.
3. **`question.errorBanner` is not in the tree at the assertion point.** When `quizState == .error` the app routes to a full-screen error view (`OOPS` / `Something went wrong` / banner / `error.retry` / `error.home`), not the in-question `errorBanner` overlay. The assertion as written ("`question.errorBanner` text is human-readable") was satisfied by reading the full-screen error label instead. The doc may want to clarify which surface to read for the error string when the state machine routes to `error`.
4. **Step ordering for log capture (per RS-01 lesson).** Launched with `launch_app_sim({args: ["--ui-test"]})` first, then `start_sim_log_cap({})` with no `captureConsole`. Verified `--ui-test` mode by hitting `http://127.0.0.1:9999/stt/connected` and getting 200.
5. **State name lives on `question.statusPill.AXValue`** (per RS-01 lesson) ŌĆö the `question.state` hidden Text probe is gone (WIP). Both the `recording` confirmation and the absence of `recording` post-wait were read from the status pill / view-tree route.

## Followups (separate from RS-02 verdict)

1. **Same auto-confirm / `processing ŌåÆ processing` bug as RS-01.** This bug now blocks two scenarios from cleanly exercising their intended path. RS-02 ends in `error` because of it, not because the auto-stop timer routed to a recoverable state. Worth filing as a discrete issue ŌĆö the auto-confirm timer should not call `resubmitAnswer` when the user did not edit, and `processing ŌåÆ processing` should either be a legal no-op or the caller should be in a different state.
2. **RS-02 hypothesis vs. mock behavior.** Either:
   - update `regression-scenarios.md` RS-02 hypothesis to "the 15 s safety timer must stop recording and either submit the auto-committed transcript or route to a recoverable state", **or**
   - add a knob to `MockElevenLabsSTTService` so `commitAndClose` can be a no-op for this scenario.
   The first option is cheaper and matches what the safety timer actually guarantees.
3. **`question.errorBanner` vs. full-screen error route.** RS-02's assert in the doc references `question.errorBanner` text. The full-screen error route does not include that identifier. Add an a11y identifier to the full-screen error label (e.g. `error.message`) and update the doc, or assert generically on any AXStaticText whose label starts with "Error".
