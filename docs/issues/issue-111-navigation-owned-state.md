## Why

Navigation is split across two incompatible models вЂ” a `quizState`-keyed root swap and a real `NavigationStack` push chain вЂ” bridged at runtime by a hidden `NotificationCenter` broadcast. One driving-loop bug already ships through the gap and a second is latent:

1. **Voice "start" over a pushed stack (shipping bug).** The "start" command calls `startNewQuiz()` directly and never fires the `.packQuizStarted` teardown, so saying "start" while Settings/OrderPack/MyPacks is pushed leaves that screen covering the fresh QuestionView (`QuizViewModel+CommandListener.swift:169`). Error-retry (`ContentView.swift:298`) shares the same latent gap.
2. **Broadcast + identity-reset bridge (latent + unmaintainable).** ContentView tears the stack down by force-recreating its identity (`navStackID = UUID()` + `.id()`) on a `.packQuizStarted` broadcast fired from one call site (`SettingsView.swift:647`). Hidden global coupling: a new listener, a second post site, or an ordering change silently breaks the #95 вЂ” custom quiz packs flow, and the mechanism is invisible to anyone reading either end alone (`ContentView.swift:186`).

Maintainability cost: 9 `startNewQuiz` call sites must each remember to pair a teardown вЂ” only 1 does вЂ” and the coupling is un-unit-testable (nav is view-level `@State`). Target rule (review dim. 4 / Top-10 item 3, [source](../research/ios-architecture-review-2026-07-18.md)): navigation is observable state owned by one object, cleared where `quizState` leaves idle вЂ” no broadcasts, no `.id()` resets.

## Scope

**In:**
- One owned route object holding the pushed-stack path (route enum / `NavigationPath`), replacing the `.packQuizStarted` broadcast + `navStackID` identity reset.
- A single teardown path all 9 `startNewQuiz` call sites converge on (incl. voice + error-retry), clearing the stack on every quiz-start.
- Migrate the 4 NavigationLinks + 1 `navigationDestination` on the single root stack (`ContentView.swift:77`) to path/route-driven pushes вЂ” the minimum for a complete, clearable teardown.
- New RS regression scenario + a unit seam on the owned route object.

**Out:**
- Sheet hosts (paywall, sign-in, mic picker, answer-confirm, quiz-settings, onboarding cover) and the `MinimizedQuizView` overlay вЂ” not on the nav stack.
- Self-contained sheet stacks (AudioDevicePicker, SourceWebView) вЂ” leave alone.
- Dead `LanguagePickerView` removal в†’ #115 вЂ” iOS 26 target / dead-code sweep.
- QuizViewModel decomposition в†’ #113 вЂ” QuizViewModel decomposition. Any visual redesign.

## Resolved design decisions

1. **Route owner вЂ” a small `NavigationModel` owned at ContentView, not QuizViewModel.** New `NavigationModel: ObservableObject` holding the pushed path (`NavigationPath` over an `AppRoute` enum: `.settings / .orderPack / .myPacks / .orderProgress`, + DEBUG `.debugLog`), created as `@StateObject` in ContentView (which already hosts the `NavigationStack`) and injected via `@EnvironmentObject`. *Why here:* the root swap stays `quizState`-driven on QuizViewModel; only the push chain needs an owner. Putting it on QuizViewModel fattens the god object #113 вЂ” QuizViewModel decomposition is shrinking; burying it in AppState overloads the service locator the review already flags (zero-`@Published` DI container). A standalone object *is* the "one object owns navigation" rule, and is exactly what #97 вЂ” CarPlay support's second scene can later observe вЂ” without building for CarPlay now.
2. **Teardown вЂ” reactive clear on the `quizState`в†’active transition, at the ContentView layer.** ContentView clears `NavigationModel.path` when `quizState` leaves `.idle` (enters `.startingQuiz`) вЂ” the same idiom as the existing root-content switch and `screenAwakeWriter` apply (research В§134-145). *Why not the two framed options:* putting it inside `startNewQuiz()` forces a VMв†’nav back-reference вЂ” the exact AppState-weak-ref coupling #113 removes вЂ” and only covers imperative callers; a wrapper every call site routes through needs 9 edits and a 10th site can forget it. Driving off the state transition every `startNewQuiz` already produces makes it structurally impossible to start a quiz without tearing the stack down вЂ” precisely the `CmdListener:169` bypass, fixed for free, for voice and error-retry alike. (Error-retry's `errorв†’startingQuiz` transition being currently rejected is a separate state-machine finding, not this issue; once it lands it enters `.startingQuiz` and inherits the clear.)
3. **Migration breadth вЂ” full within the one root stack, nothing beyond it.** All 5 pushes on `ContentView.swift:77`'s stack (Homeв†’Settings, Settingsв†’OrderPack, Settingsв†’MyPacks, Settingsв†’DebugLog, OrderPackв†’OrderProgress) move to path/route-driven. *Why full, not partial:* they share one `NavigationStack`; a half-migrated stack leaves old-style pushes the path can't clear, so teardown is incomplete and the bug returns вЂ” full migration of *this stack* is the minimum for correctness, not over-abstraction. *Why not broader:* self-contained sheet stacks and dead `LanguagePickerView` stay untouched (Minimal Footprint / simple-and-robust bar). DebugLog is DEBUG-only вЂ” folded into the sweep for consistency, not load-bearing. No multi-scene / CarPlay route coordinator now вЂ” the single `NavigationModel` extends to #97 without speculative abstraction.
4. **Test strategy вЂ” new RS + unit seam on the route object.** (a) End-to-end `RS-pack-nav-start` (`testRSPackNavStart` in `HangsUITests/Regression/RegressionTests.swift`): push the full `#95` shipping depth **Homeв†’Settingsв†’OrderPackв†’OrderProgress(delivered)** and fire "start" **from the delivered OrderProgress** via the CTA and the voice HTTP-listener path (two separate relaunch passes), asserting QuestionView is the clean visible root with no OrderProgress/OrderPack/Settings covering it вЂ” the regression that would have caught the bypass *and* proves the `isPresented` child tears down; needs new Settings/OrderPack/OrderProgress page objects (current page objects are Home/Paywall/Question/Result only). (b) Unit test on `NavigationModel`: seed a non-empty path **and `orderProgressPresented = true`**, drive `quizState` idleв†’`.startingQuiz`, assert the path empties **and the `isPresented` flag resets** вЂ” the bypass-proof property, now assertable off-sim because nav is an owned observable (was un-assertable view-level `@State`).

**Founder decision needed (non-blocking).** After a pack quiz starts, tearing the stack down means post-quiz Completion / "Back" lands on **Home**, not back in **MyPacks**. Default recorded: **reset to Home** вЂ” preserves current shipped behavior (bug-preserving) and is trivially reversible later (re-push MyPacks onto the cleared path on `.finished`). The plan executes on this default; do **not** block. Confirm interactively when convenient.

## Tasks (atomic)

Ordered, each independently committable and green on its own. Two Phase-4 resolutions of the gate notes, load-bearing for every task below:

- **Teardown invariant (gate note 1):** `NavigationModel.path` is emptied **whenever `quizState` *enters* `.startingQuiz`** вЂ” the sole quiz-start transition, for *every* predecessor. Today that is only `.idle в†’ .startingQuiz`; once [#110 вЂ” quiz state-machine enforcement](issue-110-quiz-state-machine-enforcement.md) adds `error в†’ startingQuiz` for retry, that entry inherits the clear too. The predicate is **"enters `.startingQuiz`"**, never "leaves `.idle`" вЂ” the two coincide only until #110 lands (`idle` can currently reach nothing but `.startingQuiz`, `QuizViewModel.swift:85`). Driving off the *entry* keeps teardown correct across that change and covers the `CmdListener:169` voice bypass + `ContentView:298` error-retry for free.
- **`AppRoute` shape (gate note 2):** a **value enum, no associated values** вЂ” `case settings / orderPack / myPacks` + `#if DEBUG case debugLog`; auto-`Hashable` for `NavigationStack(path:)` + `NavigationLink(value:)`. The four root pushes are rebuilt from `viewModel`/`appState` inside one `navigationDestination(for: AppRoute.self)` closure, so none needs a payload. The **OrderPackв†’OrderProgress** push is the exception the reviewer flagged: `OrderProgressView` (`@ObservedObject var viewModel`, `OrderProgressView.swift:13`) observes OrderPack's *live* `@StateObject OrderPackViewModel` (`OrderPackView.swift:13`, pushed via `navigationDestination(isPresented:)` at `OrderPackView.swift:55`) вЂ” a reference type that cannot live in a `Hashable` enum. It therefore **stays OrderPack-local `navigationDestination(isPresented:)`**. **Belt-and-braces (do NOT bet on transitive pop):** SwiftUI's dismiss of a `navigationDestination(isPresented:)` child when the *parent* path clears is community-documented as unreliable (mixing `isPresented` with `NavigationStack(path:)` may leave the `isPresented` child mounted). So its binding is **not** local `@State` вЂ” OrderPack drives it from `NavigationModel.orderProgressPresented`, which the teardown (`handleQuizStateChange`) resets in the **same** step it empties `path`. OrderProgress then dismisses deterministically regardless of the framework's transitive behaviour. Proven off-sim (A1 case d asserts the flag resets) and on-sim (T4 fires start from the delivered OrderProgress вЂ” the deepest `isPresented` point). This satisfies Resolved decision 3's actual requirement (complete teardown of *this* stack) without moving the reference-typed push into the enum.

1. **`NavigationModel` + `AppRoute` types + unit seam.** New `apps/ios-app/Hangs/Hangs/ViewModels/NavigationModel.swift`: `AppRoute` (shape above) and `NavigationModel: ObservableObject` with `@Published var path`, the belt-and-braces `@Published var orderProgressPresented = false` (the lone `navigationDestination(isPresented:)` binding, driven off this object вЂ” see T2/gate note 2), and `handleQuizStateChange(_ new: QuizState)` that iff `new == .startingQuiz` empties `path` **and** sets `orderProgressPresented = false` (a single `clearAll()` вЂ” the whole nav surface, path + isPresented child, resets atomically).
